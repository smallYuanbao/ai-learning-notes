# 意图路由

> 📅 学习日期：2026-07-14
> 🔗 关联面试题：RAG 专题 题24（召回结果与意图不匹配的改进方向）

---

## 一、什么是意图路由

意图路由是在 RAG 管道中增加一个**轻量级预处理步骤**，先识别用户问题属于哪个类别，再根据类别优化后续的检索和生成策略。

**核心作用**：
- 缩小检索范围，减少无关文档的噪声干扰
- 为不同类别的问题选择合适的 Prompt 模板或检索策略

---

## 二、我的实现方案：三级分类 + 降级兜底

### 分类体系

7 类育儿话题：

| 类别 | 种子关键词示例 |
|------|--------------|
| 常见病护理 | 发烧、咳嗽、腹泻、湿疹、感冒 |
| 喂养 | 母乳、配方奶、辅食、厌奶、奶量 |
| 睡眠 | 夜醒、哄睡、早醒、睡眠倒退 |
| 生长发育 | 身高、体重、大运动、语言发育 |
| 疫苗与体检 | 疫苗反应、接种时间、自费疫苗 |
| 日常护理 | 洗澡、脐带护理、抚触、剪指甲 |

### 三级分类策略

| 置信度 | 处理方式 | 耗时 |
|:---:|------|:---:|
| ≥ 0.9 | 直接使用关键词匹配结果 | <1ms |
| 0.8 ~ 0.9 | 关键词结果 + LLM 复核 | ~500ms |
| < 0.8 | 交给 LLM 分类 | ~500ms |

### 降级兜底

LLM 调用失败（超时/网络异常/返回格式错误）时，回退为 `general` 通用类别，不阻塞主流程。

## 三、在 RAG 管道中的位置

用户问题
→ Query Rewrite（指代消解）
→ 意图路由（关键词快速匹配 + 置信度分层）
→ 混合检索（Dense + Sparse）
→ Prompt 拼接
→ LLM 生成


当前意图路由放在**检索之前**，用于记录日志和为后续优化（如按类别缩小检索范围、动态调整 α 值）预留接口。

### 意图路由和混合检索 的顺序

- 先意图路由后混合检索，流程：用户问题 → 意图分类 → 根据类别定向检索	，缩小检索范围，减少无关文档干扰。
    - 检索效率更高，确定用户的意图后，只检索意图相关的文档子集
    - 减少噪声文档，先识别到意图，可以过滤掉大量不相关的文档
    - 可以动态调整检索策略，知道意图类别后，可以根据意图类别动态调整 α 值

- 先混合检索后意图路由，流程：用户问题 → 全库检索 → 意图分类 → 过滤结果或调整 prompt，对已召回的结果做后处理。
    - 好处是意图识别不阻塞检索，检索结果完整。
    - 如果知识库很大、类别很多，全库检索会召回大量无关类别的文档，浪费计算资源，也增加了 LLM 的噪声


**特殊处理**：如果意图路由返回的是emergency，直接跳过混合检索，并不需要混合检索给的内容而是需要直接出急救指令

## 六、主流实现方案对比

| 类型 | 方案 | 核心原理 | 优点 | 缺点 |
|------|------|---------|------|------|
| **基于规则** | 关键词匹配 | 用种子关键词与用户问题做命中统计，计算置信度 | 速度极快（<1ms），无成本，可解释性强 | 无法处理未命中关键词的模糊问题，分类粒度粗 |
| | 正则表达式 | 用正则模式匹配特定句式（如“怎么吃”、“怎么办”） | 精准匹配特定模式 | 维护成本高，覆盖面窄 |
| **基于模型** | LLM 分类 | 调用 LLM，用 Prompt 让模型直接输出分类标签 | 灵活性高，能处理模糊和复杂意图，无需维护关键词库 | 有延迟（~500ms），有 Token 成本，结果非确定性 |
| | 专用分类模型 | 用微调后的 BERT 等轻量模型做文本分类 | 速度快，成本低，效果可控 | 需要标注数据、训练和部署额外模型，工程复杂度高 |
| **混合式** | 关键词 + LLM 分层 | 先关键词快速匹配，根据置信度决定是否交给 LLM 复核（我的方案） | 兼具速度（高置信直出）和灵活性（低置信 LLM 兜底），工程成本低 | 需要设定置信度阈值，关键词库需初期人工定义 |


## 七、我的方案评价

### 方案类型

混合式：**三级分层 + 降级兜底**

### 优势

1. **工程实用性强**：兼顾了性能和灵活性。大部分常见问题（如“宝宝发烧”、“宝宝不爱吃饭”）都能被关键词命中，直接快速分类（<1ms）；少数边界问题交给 LLM，不阻塞主流程。
2. **可观测**：每个请求都能看到分类结果和置信度，方便调试和迭代关键词库。
3. **低成本**：关键词匹配零成本，LLM 调用仅在必要时触发（中低置信度），不像纯 LLM 方案那样每次请求都消耗 Token。

### 待优化点

| 优化项 | 当前状态 | 改进方向 |
|--------|:---:|---------|
| **关键词库覆盖** | 6 类，每类约 10 个种子词 | 可从知识库中自动提取高频词或用户反馈中补充新词 [实施方案](https://github.com/smallYuanbao/baby-ai/blob/refactor/python-server/python-server/docs/optimization-plans/01-%E5%85%B3%E9%94%AE%E8%AF%8D%E5%BA%93%E8%87%AA%E5%8A%A8%E6%89%A9%E5%85%85.md)|
| **置信度计算** | 基于命中关键词数量和权重 | 可引入词向量相似度辅助计算，避免“有词但无关”的误判 [实施方案](https://github.com/smallYuanbao/baby-ai/blob/refactor/python-server/python-server/docs/optimization-plans/02-%E8%AF%AD%E4%B9%89%E7%9B%B8%E4%BC%BC%E5%BA%A6%E6%A0%A1%E9%AA%8C.md) |
| **多意图处理** | 不支持 | 用户问题可能同时涉及“发烧”和“喂养”，可改为多标签分类或返回 top-2 [实施方案](https://github.com/smallYuanbao/baby-ai/blob/refactor/python-server/python-server/docs/optimization-plans/03-%E5%A4%9A%E6%84%8F%E5%9B%BE%E5%A4%84%E7%90%86.md)  |
| **类别数量** | 固定 6 类 + 兜底 | 业务扩展时可手动增加类别，或用 LLM 自动发现新类别  [实施方案](https://github.com/smallYuanbao/baby-ai/blob/refactor/python-server/python-server/docs/optimization-plans/04-%E8%87%AA%E5%8A%A8%E5%8F%91%E7%8E%B0%E6%96%B0%E7%B1%BB%E5%88%AB.md)|

### 面试回答模板

> “我采用了关键词 + LLM 的混合方案。先用 6 类育儿种子关键词做快速匹配，按命中率计算置信度。置信度 ≥0.9 直接用关键词结果，0.8-0.9 调用 LLM 复核，低于 0.8 完全交给 LLM 分类。LLM 调用失败时降级为 general 兜底。这样设计既保证了常见问题的毫秒级响应，又能覆盖模糊问题，同时不会因为 LLM 异常导致整个请求失败。”
