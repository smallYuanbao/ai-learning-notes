## 题1：Prompt 设计与优化的技巧及实例

**我的回答**：

我把 Prompt 设计分成两层：**先定调，再执行**。

**第一层是定调——明确角色定位。** 
我不会只说“你是育儿助手”，而是具体到“拥有15年临床经验的儿科主任医师，风格温暖但不啰嗦，专业但不吓人”。角色越具体，LLM 的语气、用词、甚至信息组织方式都会自动对齐这个人设，比泛泛的角色靠谱得多。

**第二层是执行——写清楚指令 + 给模型思考时间。** 

**写清楚指令，我用了四个技巧：**
- **分隔符**：用 `<参考资料>` 和 `<用户问题>` 的 XML 标签，把知识和任务区严格分开，既防混淆也防注入。
- **结构化输出**：强制回答按 🏷️结论、📖原理、✅做法、🚨就医指征、🩺免责 五个板块输出，也可以要求 JSON、HTML、表格等格式，方便前端解析。
- **条件检查**：让 LLM 先判断参考资料是否充分，不充分就诚实说明，而不是硬编。育儿场景下这关乎安全。
- **Few-shot 示例**：从知识库里选了 2 条高质量 Q&A（一条疾病处理、一条日常护理），结果模型的格式和语气几乎完全对齐，仅两条就定义了整个回答规范。

**给模型思考时间，核心就是思维链（CoT）。** 
强制模型先写出推理步骤，再给最终结论，而不是跳步直接出答案。常见做法有两种：
- **零样本思维链**：只在问题末尾加一句“请一步步思考，再给出最终答案”，简单但有效。
- **少样本思维链**：提供几套“问题→分步推理→答案”的模板，让模型模仿你的思考格式。

我在项目里把这招落地成三步：先不看资料给出初步回答 → 再读资料判断初答是否准确 → 最后结合资料生成最终回答。这本质上就是给模型预设了一条推理链，让它先自己思考，再对照参考资料修正，从而避免被资料带偏或跳过关键判断。

最后，Prompt 不是一次写成的，我会用一批测试问题跑新旧版本，对比准确性、结构完整性和幻觉率，像迭代代码一样迭代 Prompt。

**项目关联**：
- baby-ai 项目的 RAG 管道中，`build_prompt` 函数落地了完整优化策略。
- 核心实践：XML 分隔符、五板块结构化输出、条件检查、2 条 Few-shot 示例、三步思维链推理。

**关键词**：
角色定位、分隔符、结构化输出、Few-shot、条件检查、思维链（CoT）、零样本/少样本、Prompt 迭代

## 题4：Few-shot Learning 的定义及 Zero-shot、One-shot、Few-shot 的差异

**我的回答**：
Few-shot、One-shot、Zero-shot 是提示词工程中的**示例提供策略**，本质上属于**上下文学习**的范畴。核心思想是：不更新模型参数，只通过 Prompt 中给出的参考答案，让 LLM 模仿着去做处理。

它们的核心区别就是**给模型的示例数量不同**：

- **Zero-shot（零样本）**：不给任何示例，纯靠指令让模型理解任务。比如最早期那版 Prompt，只说了“你是一个育儿专家助手”，没给任何 Q&A 样例，模型全靠指令来推测格式和语气。优点是灵活、占用 token 少，缺点是输出格式不可控。
- **One-shot（单样本）**：给 1 个示例，模型照着这个唯一的样本模仿。比如只给“疫苗发烧”那一条 Q&A。优点是格式明确，缺点是可能过拟合这条样本的风格，遇到不同类型的问题时适配性差。
- **Few-shot（少样本）**：给 2-5 个示例，模型从多个样本中归纳规律。我在项目里最终选了这个策略——给了两条示例，一条疾病处理、一条日常护理。两个场景足以让模型学到通用的格式和语气，又不会吃掉太多上下文窗口。

**选型逻辑**：Zero-shot 适合任务简单、格式要求不高的场景；One-shot 适合格式固定但场景单一的任务；Few-shot 适合需要一定泛化能力、但又不想让示例占用太多 token 的场景。

**项目关联**：
- baby-ai 项目的 RAG 管道中，`build_prompt` 函数落地了完整优化策略。
- 核心实践：2 条 Few-shot 示例。

**关键词**：
示例提供策略、上下文学习、Few-shot、Zero-shot、One-shot

**面试官可能追问**：

**Q1：你项目里为什么选 Few-shot 而不是 One-shot 或 Zero-shot？**
🟢 当前状态：能讲清

- Zero-shot 我试过——最早那版 Prompt 就是 Zero-shot，没给任何示例。结果是格式不可控，有时 LLM 把回答写成一整段，有时缺少某几个固定板块，前端解析很被动。
- One-shot 只给一个示例的问题在于过拟合——如果那条示例是“疫苗发烧”，当用户问“辅食加盐”时，模型容易机械套用“发烧”场景的框架，而不是灵活适配新问题类型。
- Few-shot 用两条示例（一条疾病处理、一条日常护理），模型能从两个不同场景中归纳出通用格式，同时不会因为示例太多吃掉上下文窗口。这是我经过实际对比后选的方案。

**Q2：Few-shot 示例你选了 2 条，怎么确定选 2 条而不是 3 条或 5 条？**
🟡 当前状态：概念可讲，部分有实验

- 示例越多，占用的上下文窗口越多，留给真实参考资料和用户问题的 token 空间就越少。我的知识库文档有时候很长，需要给它们留足空间。
- 2 条刚好覆盖两个核心场景（疾病处理 + 日常护理），从实验效果看，模型已经能归纳出通用的输出格式。增加到 3 条以上，边际收益递减——格式提升不明显，但 token 消耗线性增加。
- 判断方法：准备一个固定的测试问题集（5-10 条），分别用 1/2/3/5 条示例各跑一遍，看回答的格式完整性和准确性。当增加示例不再带来明显提升时，就是最优数量。

**Q3：你的 Few-shot 示例放在 Prompt 的什么位置？放开头和放末尾有区别吗？**
🟡 当前状态：能讲清概念

- 我放在 System Prompt 之后、用户问题之前。这样模型先理解自己的角色和任务，再看示例模仿格式，最后处理真实的用户问题。
- LLM 对 Prompt 的开头和结尾更敏感（即“首因效应”和“近因效应”）。如果示例很长，放在中间可能被模型“忽略”一部分。我的示例很短（每条 4-5 行），放在指令和用户问题之间刚好。
- 如果示例很多（超过 5 条），建议把最重要、最有代表性的示例放在最后一条（离用户问题最近），模型会优先参考它。

**Q4：如果用户问了一个和你的 Few-shot 示例完全不相关的问题，示例还有用吗？**
🟢 当前状态：能讲清

- 有用，但作用的维度不同。Few-shot 示例解决的核心问题是**格式和语气的定义**，而不是**具体内容的填充**。
- 即使用户问的是我示例中没有覆盖的话题（比如“宝宝鼻塞怎么办”），模型仍然会从示例中学到“先给结论→讲原理→分步操作→就医指征→免责声明”这个框架，然后把新话题的内容填进去。
- 这就像给了模型一个“作文模板”——题目可以换，但结构不变。如果我希望模型学到的是内容层面的知识（比如特定疾病的诊疗规范），那应该放到 RAG 的知识库里，而不是靠 Few-shot 示例来覆盖。

**Q5：Few-shot 和思维链（CoT）你是怎么配合用的？**
🟡 当前状态：能讲清概念，有项目落地

- Few-shot 解决“模仿谁”的问题，CoT 解决“要不要展示推理过程”的问题，两者可以叠加。
- 我的项目里就是这样叠加的：两条 Few-shot 示例里，每条都展示了推理结构（先给结论→讲原理→分步骤→指征→免责），然后 Prompt 末尾用三步推理链（先不看资料回答→再看资料验证→最后生成）要求模型在执行时也走同样的推理流程。
- 效果是：模型既学会了“格式长什么样”（Few-shot），又被强制要求“一步步想”（CoT），两个机制互不冲突，反而互补

## 题9：temperature 与 top_p 参数的作用及选型方法

**我的理解**：
LLM 生成下一个 token 时，面对的是一个固定不变的词表（比如 DeepSeek 约 10 万个 token）。这个集合本身不会因参数调整而增删任何一个词。
- **temperature** 不改变集合，只改变概率分布的形状：值越低，分布越尖锐，高质量 token 独占概率；值越高，分布越平坦，更多 token 分摊概率。
- **top_p** 在这个已调整的分布上画一条截止线——从概率最高的 token 开始累加，累到累积概率达到 p 就停，线以下的 token 直接排除。

两者配合的逻辑是：temperature 先决定分布形态，top_p 再基于这个形态圈定候选范围。如果 temperature 太高把分布拉平了，即使 top_p 设得很低，候选池里也可能混进了原本不该出现的词——这就是为什么 temperature=1.5 时即使 top_p 不高，输出仍然可能不可靠。

**我的回答**：

temperature 和 top_p 都是控制 LLM 输出随机性的参数，但作用机制不同：
- **temperature**：调节概率分布的“尖锐程度”。值越低，高概率 token 的优势越明显，输出越确定；值越高，低概率 token 被选中的概率越大，输出越有“创造性”但也可能胡编。
- **top_p（核采样）**：设定累积概率阈值，只从这个阈值内的 token 集合里采样。值越低，候选词越少，输出越保守；值越高，候选词越多，表达越丰富。

我用同一个问题“宝宝发烧 38.5 度怎么办？”做了两组对照实验：

1. **固定 temperature=0.7，对比 top_p**：
   - top_p=0.1：回答保守，用词集中在高频表达，结构是线性叙述。
   - top_p=0.9：回答更丰富，自动生成了表格和颜色分级，语气也更有层次。

2. **固定 top_p=0.9，对比 temperature**：
   - 0.0：回答高度确定、可复现，适合需要准确性的场景。
   - 0.7：兼顾可靠性和表达丰富度。
   - 1.5：出现幻觉式引用（编造“参考资料 2 和 3”），不适合生产环境。

**项目选型**：育儿助手使用 temperature=0.7 + top_p=0.9，在可靠性和回答多样性之间取得平衡。对于评分类子任务（如 LLM 自评打分），降到 temperature=0 以确保一致性和可复现性。

**关键词**：temperature、top_p、核采样、输出可控性、随机性调节

**面试官可能追问**：

**Q1：temperature 和 top_p 同时调 vs 只调一个，你的选型逻辑是什么？**

🟢 当前状态：能讲清

我的策略是：**temperature 是主旋钮，top_p 是噪声过滤器，默认只调主旋钮。**

- **默认只调 temperature**：它直接控制概率分布的形状——想让回答更确定就调低，想让它更有创造性就调高。效果直观可预期，大多数场景只调这一个就够用。
- **什么时候引入 top_p**：当 temperature 已经调到合适的范围（比如 0.7-0.8），回答的多样性够了，但偶尔出现一些明显不靠谱的表达（用词生僻、逻辑跳跃）。这时候不是降 temperature——降了会牺牲已经调好的多样性，而是**降一点 top_p（如从 1.0 降到 0.9）来截掉尾部的低概率噪声**。
- **降 top_p 的代价**：确实会让候选词变少，多样性会微降。但它砍掉的主要是那些大概率产生幻觉的词，多样性的核心来源（中高概率区间的 token）仍然被保留。这不是“多样性 vs 质量”的二选一，而是“用微小的多样性换质量的大幅提升”。
- **口诀**：temperature 决定“敢不敢选”，top_p 决定“选的范围有多大”。敢选是前提，范围做微调。

**Q2：你项目中除了 temperature 和 top_p，还用过其他控制输出的参数吗？**

🟡 当前状态：概念可讲，部分未落地

- **已落地使用的**：
  - `max_tokens`：控制回答长度上限，避免 Token 浪费和无限生成。
  - `stop`：指定停止词，如结构化输出的某个固定板块后截断，防止模型画蛇添足。
- **了解但未在项目中使用的**：
  - `top_k`：和 top_p 一样用于限制候选 token 范围，但它是固定数量 k 做硬截断。我的场景下 temperature + top_p 已经够用，top_k 的固定截断反而可能在概率分布均匀时误杀合理候选，或在概率高度集中时放进噪声。
  - `frequency_penalty`：抑制重复用词，更适合创意写作。育儿问答中重复用词不是主要问题。
  - `presence_penalty`：鼓励引入新话题，同样更适合多话题覆盖场景。
- **选型原则**：参数不是越多越好，每个参数都有代价。我优先用 Prompt 工程（指令 + 示例）控制行为，参数只是兜底手段。如果调 Prompt 能解决的问题，就不引入新参数。

**Q3：temperature=1.5 时出现幻觉，为什么高 temperature 更容易导致幻觉？**

🟢 当前状态：能讲清

- LLM 生成每个 token 时面对固定词表，模型算出每个词的概率分布后，高 temperature 把这个分布拉平了——原来概率 0.01% 的词可能被抬到 1%，原来概率 80% 的词可能降到 30%。
- 被抬高的“低概率词”通常是不太可能出现在当前上下文里的词——比如编号（“参考资料 2 和 3”）、生僻术语、逻辑跳跃的表达。
- 一旦某一步选错了词，后续 token 都在这个错误基础上继续生成，错上加错，最终表现为“看起来更专业但实际在编造”的幻觉。

**Q4：如果你的产品需要“创造力”和“准确性”同时兼顾，你会怎么设计？**
🟡 当前状态：概念可讲，待落地

- 双阶段架构：生成阶段用较高 temperature（0.8-0.9）产出多个候选回答，保证覆盖足够多的可能性；筛选阶段用另一个 LLM 调用（temperature=0）对候选回答做评估，选最准确的那个。
- 这种“生成-评估”架构比单纯调参数更可靠，能应对更复杂的需求。
- 育儿助手目前是简化版：生成回答用 temperature=0.7，评分子任务用 temperature=0，本质上是同一思路的轻量落地。
