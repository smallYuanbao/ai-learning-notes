## Introduction & Guidelines for Prompting 笔记
设计高效的Prompt
### 原则一：写清楚指令
#### 1、使用分隔符清晰地表示输入的不同部分
用 ###、"""、---、XML标签等分隔符，把指令、背景材料、用户输入清楚的分开，避免LLM混淆。
#### 2、寻求结构化的输出
要求模型以特定格式输出，比如JSON、HTML、MarkDown表格等，方便程序解析
#### 3、要求模型检查是否满足
给模型设定边界，让它先判断”这个问题我能不能回答“，不能答就直接说，不要硬编。
#### 4、提供少量示例（few-shot）
提供几个简单的示例，让AI模仿风格和格式。这是提升输出质量的最直接的方式。
### 原则二：给模型思考时间
#### 1、指定完成任务所需的步骤
把复杂任务拆成步骤，让模型按顺序执行，避免跳步或遗漏
#### 2、指导模型在下结论前找出一个自己的解法
让模型先独立推理，再给结论，避免被参考资料带偏或直接跳过思考。

## 代价与改进方向

新 Prompt 让模型输出了完整的思考过程（步骤一/二/三），虽然增强了可解释性，
但直接展示给用户会造成信息冗余。
新 Prompt 让 LLM 输出了完整的思考过程，虽然增强了可解释性，但消耗了额外的输出 Token。

**改进方案**：
- 在 `call_deepseek` 函数中加入解析逻辑，用正则提取从 "🏷️" 开始的内容作为最终回答
- 或改为要求 LLM 输出 JSON（thinking + answer 两个字段），从协议层面解耦思考过程和回答

**优化策略（面试时可展开讲）**：
1. **内化思考**：要求 LLM 在内部推理但不输出，只返回最终回答，可大幅减少 Token。
   - Prompt 里要求“请在你的内部进行推理，但最终回答里不要展示推理过程，只输出从 🏷️ 开始的结构化回答”
   - 模型仍会在内部进行推理（消耗 prompt/completion 的隐性算力），但不生成思考文字，大幅减少输出 Token
2. **两阶段调用**：先让 LLM 用极短输出判断资料是否充分，再决定是否需要深度推理。
    - 第一次调用：让模型输出简短判断（如“参考资料是否足够：是/否”），10 个 Token 以内。第二次调用：根据判断结果，选择不同的 Prompt 生成最终回答
    - 简单问题省掉大量思考 Token，复杂问题保留深度推理，按需消耗
3. **高频问题缓存**：对常见问题缓存最终回答，直接绕过 LLM 调用，Token 消耗降为零。
    - 对于高频问题（如“宝宝发烧怎么办”），把第一次的优质回答缓存到 Redis，后续相同问题直接返回缓存，不走 LLM
    - Token 消耗降为零，同时延迟降到毫秒级

## 综合 Prompt 实验

**问题**：宝宝发烧 38 度需要吃药吗？

**旧 Prompt 回答**：（贴关键差异）
- 直接给出答案，没有展示推理过程
- 回答结构虽接近要求，但格式不严格，没有明确的五个板块标识
- 语气温暖但缺乏量化边界，对参考资料的使用不明显

**新 Prompt 回答**：（贴关键差异）
- 明确展示步骤一、二、三的完整推理链
- 步骤二中模型主动指出“参考资料主要针对疫苗后发烧，信息不完全”
- 最终回答严格遵循五个板块，并且与 Few-shot 示例的格式高度一致
- 就医指征部分自动生成了表格，结构化程度显著提升

**发现**：
1. **思考链让推理过程透明化**：旧 Prompt 直接给答案，新 Prompt 展示了模型如何评估资料质量、发现信息缺口、再结合知识补充，可靠性大幅提升。
2. **分隔符和条件检查有效抑制了幻觉**：模型在步骤二中主动识别到参考资料信息不完全，并诚实说明，而不是强行套用不匹配的建议。
3. **Few-shot 的格式迁移效果超预期**：新回答的表格、emoji 使用、五板块结构几乎完全克隆了两个示例的风格，2 条高质量示例就足以定义整个回答规范。
4. **代价是回答变长且包含冗余信息**：思考步骤直接展示给用户会造成阅读负担，引出后续优化需求（后端正则提取或 JSON 分离）。
