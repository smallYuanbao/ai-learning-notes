## 题1：简述 RAG 的定义及核心工作流程 + 局限性

**我的回答**

**定义**：检索增强生成，它是一种让大模型LLM通过查阅外部知识库来回答问题技术。

**流程**: 文档解析 -》数据提取-》数据清洗-》文档分块-》向量化（Embedding）嵌入-》混合检索-》Reranker 精排-》LLM

**局限性**：
1. 垃圾进，垃圾出，RAG的效果高度依赖第一步的检索质量
  - 分块切分难题，如果PDF文本切分时把一段完整的话从中间切断，语义会受损。
  - 多模态能力差，传统RAG极难处理PDF中复杂的表格、流程图、扫描件。如果fitz提取出的文本排版错乱，大模型读的就是一堆乱码
  - 向量检索局限，常用的向量相似度只能理解字面或相近意思，面对“除了A之外还有谁”这种包含取反、逻辑推理的提问时常常失效

2. 生成阶段局限
  - 断章取义：检索出的片段往往零散，大模型缺乏全书的宏观背景，容易给出片面回答
  - 整合失败：当检索回来的多份文档相互冲突，大模型很难自主判断选哪一个，甚至会把两份资料胡乱缝合
  - 固执己见：如果大模型自身的预训练记忆非常强大，即使它喂给它正确的外部资料，它有时仍会忽略提示词，继续按照自己的“偏见”胡说八道

3. 性能与成本权衡
  - 首字延迟高：整个流程需要查数据库、做重排，再把大段文本拼接Prompt传给模型，导致用户等待回答的时间变长。

  - 上下文窗口焦虑：检索出来的文档越多，消耗的Token越多，API费用直线上升。

**项目关联**：
baby-ai 完整落地了这条链路

**关键词**：RAG、检索增强生成、文档解析、分块、Embedding、混合检索、Rerank、幻觉


## 题2：RAG 中 Rerank 的含义及实现方式

**我的回答**：

**定义**：Rerank（重排序）是在混合检索之后、Prompt 拼接之前，对候选文档做二次精排的环节。它用更高精度的模型对每一对 (query, doc) 做全文本交叉注意力打分，选出真正相关的 top-K 文档注入 LLM。

**为什么需要 Rerank**：混合检索使用的 Bi-Encoder 是“隔空比手势”——query 和 doc 各自独立编码成向量，从未在同一语境下交互过，只能通过向量距离猜相关性。Rerank 使用的 Cross-Encoder 是“面对面审”——把 query 和 doc 拼在一起做全 token 交叉注意力，真正理解两者的匹配程度。粗筛召回的是“可能相关”，精筛选出的是“确实相关”。

**我的实现方式**：

1. **选型**：BGE Reranker v2-m3，本地 FastAPI 微服务部署。输入格式是把 query 和每个候选 doc 拼成 `[query, doc]` 对，输出是相关性分数（分数越高越相关）。

2. **在 RAG 管道中的位置**：

用户问题
  → Query Rewrite（指代消解）
  → 意图路由（6类分类）
  → 混合检索（Dense + Sparse，粗筛候选 100 条）
  → **Rerank 精排（BGE Reranker v2-m3，选出 top-5）**
  → Prompt 拼接
  → LLM 生成
3. **三级降级策略**：这是我最核心的工程亮点。
- 首选：BGE Reranker v2-m3，精度最高。
- 备选：DeepSeek 排序，调用 LLM 对候选文档逐一打分。当 BGE 服务不可用时自动切换。
- 兜底：原始向量检索结果，直接取 top-K 使用。当 LLM 调用也失败时触发。

这套设计保证了任何子服务异常下系统都不会崩溃，用户仍能获得回答，只是可能缺少重排序的精度提升。

**项目关联**：
- 代码位置：`reranker/` 目录下的 FastAPI 服务
- 集成点：混合检索返回候选文档后，调用 Reranker 精排，取 top-5 注入 Prompt
- 核心实践：BGE Reranker v2-m3 + 三级降级

**关键词**：Rerank、重排序、Bi-Encoder、Cross-Encoder、BGE Reranker、降级策略

## 题3：混合检索的定义及在大模型应用中的核心作用

**我的回答**

**定义**：
混合检索是将 **Dense（稠密向量检索）** 和 **Sparse（稀疏关键词检索）** 结合到一起的检索方案。Dense 擅长理解同义表达（“发烧”≈“发热”），Sparse 擅长精确匹配专有名词（“对乙酰氨基酚”）。两者互补，解决了纯 Dense 或纯 Sparse 各自召回不全的问题。

**核心作用**：
在大模型应用中，混合检索的核心作用是**从源头减少因信息缺失而导致的 LLM 幻觉**。传统关键词匹配无法理解语义变体，纯向量检索又容易漏掉精确的关键词——比如用户问“对乙酰氨基酚怎么吃”，纯 Dense 可能只召回“退烧药”相关文档，不包含精确剂量信息。Sparse 则能精准命中“对乙酰氨基酚”这个词，召回包含剂量表的文档。两者结合后，LLM 既有护理指南又有精确剂量，回答既全面又准确。

**项目关联**：
在我的育儿助手项目里，我用 bge-m3 一个模型同时输出 Dense 和 Sparse 两种向量，省去了额外部署 BM25 引擎的成本。融合策略上，先用线性加权（α=0.7）快速验证，后通过 24 条评估数据对比发现 α=0.5 忠实度最优，已修正默认值。实测中，对于”宝宝发烧对乙酰氨基酚怎么吃”这样的复合查询，混合检索能将语义相关的护理指南和精确匹配的剂量表同时返回，显著提升了检索命中率。

**关键词**：
混合检索、Hybrid Retrieval、Dense、Sparse、bge-m3、RRF、线性加权

## 题4：详解 RAG 的完整工作流程

> 与题1互补：题1偏"是什么 + 有什么坑"，本题偏"每一步为什么不能省 + 我的项目实践"

**我的回答**：

RAG 完整链路分 **离线准备** 和 **在线检索** 两个阶段，共 7 个核心步骤：

### 离线准备（文档入库）

**1. 文档解析** — 把 PDF/Word/HTML 等格式各异的文档统一提取为纯文本。
- **为什么不能省**：不解析 Embedding 完全失效——PDF 的二进制字节流无法向量化。
- **我的实践**：按 MIME type 分发，文本型 PDF 用 PyMuPDF，扫描件走 OCR，统一输出 Markdown。

**2. 数据清洗** — 去除噪声、统一编码、修复乱码。
- **为什么不能省**：页眉页脚、广告、版权声明混入向量库，检索时噪声文档挤占 Top-K。
- **我的实践**：正则剔除控制字符 + chardet 自动编码检测强转 UTF-8 + Jaccard 去重。

**3. 文档分块** — 把长文档切为大小适中的片段。
- **为什么不能省**：Embedding 模型有输入长度限制（bge-m3 最大 8192 tokens），超过直接截断丢信息。且块太大检索定位不准，太小丢失上下文。
- **我的实践**：Q&A 数据集天然分段不需额外切分；长篇文档用递归分割，chunk_size=512，overlap=50。

**4. 向量化 + 入库** — 分块文本经 Embedding 模型转为向量，存入向量数据库。
- **为什么不能省**：这是 RAG 的核心——只有转成向量才能在数学空间里算"语义像不像"。
- **我的实践**：bge-m3 → 1024 维向量 → ChromaDB，同时写入元数据（category、source）支持后续过滤。

### 在线检索（用户提问）

**5. Query Rewrite + 意图路由** — 对用户问题做指代消解和分类。
- **为什么不能省**：用户问"他需要吃药吗"，不消解"他"→"宝宝"，Sparse 检索完全失效。不分意图全库盲搜，噪声多延迟高。
- **我的实践**：指代消解用 LLM + 对话历史；意图路由用 6 类关键词 + LLM 分类降级。

**6. 混合检索 + Rerank 精排** — Dense + Sparse 双路召回，再精排取 Top-K。
- **为什么不能省**：纯 Dense 漏专有名词（"对乙酰氨基酚"），纯 Sparse 不懂同义（"发烧"≠"发热"）。Rerank 做全文本交叉注意力精筛，把"可能相关"变成"确实相关"。
- **我的实践**：bge-m3 单模型双输出 → 线性加权 + RRF → BGE Reranker v2-m3 精排 → Top-5。

**7. Prompt 拼接 + LLM 生成** — 把检索到的文档注入 Prompt，LLM 基于资料生成回答。
- **为什么不能省**：没有检索文档就是纯 LLM 凭记忆回答，幻觉不可控。
- **我的实践**：XML 分隔符隔离知识和问题 + 条件检查（资料不足时诚实说明）+ 结构化输出固定板块。

**关键词**：文档解析、数据清洗、文档分块、向量化、混合检索、Rerank、Query Rewrite、意图路由


## 题5：RAG 应用中，优化检索精度的数据清洗与预处理方法

🔴 待整理 → 📅 文档解析与分块学习完成后
- 已掌握：数据清洗的定义、在 RAG 五步流程中的位置、医疗数据集的筛选过滤实践
- 待补充：PDF/Word/网页等不同格式的常见噪声类型及清洗手段

**我的回答**

数据清洗定义：原始文档含噪声（页眉页脚、广告、乱码）

数据清洗的方式：

1. 文本标准化，这一步的目的是为了降低文本的特征离散度。如果用户输入”RAG“，而文档里写的是"rag"，不标准化就会导致向量相似度下降
  - 统一编码格式（UTF-8）
  - 转化为小写形式，
  - 规范数字表达式
2. 噪声过滤，这一步的目的是为了纯净化内容，把广告、版权信息做进向量库的话，影响向量空间。
  - 移除无关符号，针对页面、页脚、版权声明、以及社交媒体文末的”点赞、关注、转发“。
  - 删除重复段落/句子
  - 过滤过短或无意义的内容片段
  - 处理乱码文本区域

3. 结构化解析，这一步是为了恢复语义脉络，RAG的核心是分块，而分块的依据取决于这一步理出来的”骨架“
  - 提取标题层级关系
  - 分离正文与元数据字段
  - 解析表格、列表等复杂数据结构
  - 保留原文档语义边界

**为什么对RAG关键**
数据清洗在 RAG 管道中处于文档解析之后、分块之前。如果跳过这一步，噪声入向量库 → 检索时噪声挤占 Top-K → LLM 拿到脏数据 → 回答质量从源头崩塌

**项目关联：**baby-ai 在数据预处理阶段对医疗 Q&A 数据集做了筛选过滤——剔除空洞回复（"好的"、"收到"）、去重、chunk_size=512 分块后保留语义完整的分块。

**关键词：**
数据清洗、文本标准化、噪声过滤、结构化解析、UTF-8、Jaccard 去重、语义边界


## 题6：查询扩展的定义及在 RAG 中的应用价值

**我的回答**


**定义：**查询扩展是一种经典的信息检索增强技术。在信息检索系统执行之前，通过添加、替换或重写用户原始输入的查询词，将其转化为一组语义更丰富、表达更准确、范围更全面的新查询组合技术。


**应用价值**

1. 攻克“词汇错配”，大幅提升检索召回率
  应用价值：查询扩展利用大模型将口语化的提问，自动扩展出专业术语、行业同义词或关联概念。通过“多路径召回”，确保那些隐藏在知识库深处，字面完全不同的硬核文档能够被精准捞出。

2. 分解复杂多意图查询

  应用价值：查询扩展可以讲一个复杂的长查询，拆解并扩展为多个独立的子查询（例如：子查询1“A公司去年净利润财务报表”；子查询2“B公司核心业务线介绍”）。系统并发去向量数据库检索这些子查询，最后合并结果，完美解决了大模型单次检索视野狭窄的局限。

3. 解决多轮对话的上下文指代消解
  - 痛点：在 RAG 客服或聊天机器人场景中，用户喜欢用代词提问。
  轮次1：“北京今天天气怎么样？”轮次2：“那那儿明天呢？”（向量数据库根本无法理解“那儿”是什么意思）。
  应用价值：查询扩展会结合历史多轮对话的Context，将轮次2的查询自动补全并重写为：“北京明天天气预报”。将无状态的单次检索转为化具备上下文感知的精准检索。

4. 降低向量检索对“未见专有名词”的语义钝化

应用价值：查询扩展可以在前置阶段结合本地维护的专有名词知识库，或者让大模型根据上下文解释这个缩写，将缩写的全称、背景定义作为扩展词一并喂给向量数据库，变相增强了Embedding模型的语义识别边界

5. 激活“生成式检索的”针尖对齐

应用价值：查询扩展的一种高阶形态叫**HyDE**(假设性文档嵌入)。它让大模型在检索钱，凭空先瞎编一个”假想的完美答案“，然后，系统拿这个大模型生成的”假答案向量“去数据库里查”真实答案“。因为”答案和答案“在向量空间中的分布极其的近，这种扩展方式能实现恐怖级的语义对齐精度。

6. 消除用户表达的噪声和歧义

应用价值：查询扩展的重写模块充当了智能纠错与去噪器。它提取出用户的真实检索意图，删除无关的干扰Token，输入给向量数据库一个极其干净、高价值的特征向量，避免了无效算力的浪费。



## 题7：自查询的含义及在 RAG 中的必要性

🟢 已完成

**我的回答**：

自查询是指 LLM 在检索之前，先对用户问题进行分析和理解，判断问题的类型或领域，再根据这个判断决定检索策略。本质上就是“先理解问题，再决定怎么搜”。

在 RAG 系统中，自查询的必要性体现在：如果用户问“对乙酰氨基酚怎么吃”，知识库里有育儿护理、药品说明、成人用药等多个类别的文档。如果不做自查询直接全库检索，可能召回成人用药剂量，而不是儿科剂量。自查询先识别出这是“儿科用药”类问题，再定向检索儿科相关文档，能显著提升召回精度。

**我的项目实践**：我实现了基于意图路由的自查询——先用 6 类育儿关键词快速匹配，高置信度（≥0.9）直接定向检索对应类别；中低置信度交给 LLM 分类。这样常见问题毫秒级响应，模糊问题也不漏掉。LLM 调用失败时降级为 general，不会阻塞主流程。

**和混合检索的关系**：自查询是“检索前缩小范围”，混合检索是“检索时多路互补”，两者可以叠加使用。

**关键词**：自查询、意图路由、分类检索、定向检索



## 题9：RAG 调优后效果评估的方法及实际应用中的评估标准

**我的回答**

## 题10：RAG 中分块的定义及分块的必要性

🟢 已完成理论，实操待文件上传落地后补充

**已掌握**：
- 分块的本质（按语义或规则将长文本拆解为小单元）
- 分块的三大必要性：LLM 上下文窗口有限、检索需要合适粒度、Embedding 模型有输入长度限制
- 七种分块策略的核心思想和适用场景

**待补充**：
- 结合项目场景说明 Q&A 数据集不需要额外分块 vs 长篇育儿指南需要分块的具体例子
- 📅 文件上传功能迁移后（7/16）补充项目实操部分

**我的回答**：

**定义**：分块是将经过解析的文档按照一定规则拆分为大小适中的文本片段的过程。这些片段是 RAG 系统中向量检索和 LLM 输入的基本单位。

**为什么必须分块**：
1. **LLM 上下文窗口有限**：即使现代模型支持 128K tokens，但输入过长会导致生成质量下降、延迟增加、成本上升。分块能控制每次检索注入的文本量。
2. **检索精度需要合适粒度**：如果块太大，一篇文档可能只有 2-3 个分块，检索时难以精确定位到相关段落；块太小，语义碎片化，丢失上下文。
3. **Embedding 模型有输入长度限制**：如 m3e-base 仅支持 512 tokens，bge-m3 支持 8192 tokens。超过限制的内容会被截断，信息丢失。

**常见分块策略有七种**（固定大小、基于句子、基于段落、语义、递归、滑动窗口、层次化），各自的差异和适用场景在题11中展开。

**项目关联**：我当前的知识库使用的是预处理好的结构化 Q&A 数据集，每条数据本身就是一个完整的分块，不需要额外切割。文件上传功能（计划中）将涉及对用户上传的 PDF/Word 等文档进行解析和分块，届时我会选择固定大小分块 + 重叠作为首期方案，兼顾实现效率和检索质量。

**关键词**：分块、chunking、chunk_size、overlap、上下文窗口

---

**面试官可能追问**：

**Q1：你的知识库用的是 Q&A 数据集，但如果用户上传了一篇 10000 字的育儿 PDF，你会用哪种分块策略？为什么？**

🟡 当前状态：能讲清选型逻辑，代码落地后补充实操

> “首期我会选固定大小分块 + 重叠，比如 chunk_size=512 tokens、overlap=50 tokens。原因是实现简单、行为可预测，能快速跑通文件上传到检索的完整链路。后续如果发现检索效果不理想——比如答案经常被截断或上下文不连贯——我会升级为递归分割，先按标题切分，再按段落，最后字符兜底，兼顾语义完整性和尺寸控制。如果再进一步，可以用语义分块，用 Embedding 相似度自动识别主题边界。”

**Q2：chunk_size 设 512 和 1024 有什么区别？你打算怎么确定最优值？**

🟡 当前状态：能讲清理论差异，定量数据待产出

> “chunk_size 越大，单个分块包含的上下文越完整，但检索精度会下降——因为一个块里混入了太多信息，向量相似度匹配时很难精确定位到答案所在的具体段落。chunk_size 越小，检索越精准，但太小的块可能丢失必要的上下文。确定最优值需要通过实验：我会用 256/512/1024 三种 chunk_size 各跑一遍评估脚本，看检索命中率和 LLM 回答准确率的差异，选综合表现最好的。这部分实验计划在评估阶段（7/18-19）完成。”

**Q3：overlap 设置有什么讲究？设 0 会有什么问题？**

🟢 当前状态：能讲清

> “overlap 是相邻分块之间的重叠区域，比如 overlap=50 就是前一个分块的最后 50 个 token 会在下一个分块开头重复出现。它的作用是防止关键信息恰好落在分块的边界处被截断。设 overlap=0 的问题：比如用户问‘对乙酰氨基酚的剂量’，而‘对乙酰氨基酚’和‘剂量’这两个词恰好被切到了两个不同的分块里，导致两个分块的向量都不足以完整回答这个问题，检索时可能两个都排不上 top-K。一般 overlap 设为 chunk_size 的 10%-20%。”

**Q4：层次化分块和普通分块的区别是什么？什么场景下值得用？**

🟢 当前状态：能讲清

> “普通分块是单层的，所有分块大小相近、地位平等。层次化分块是多层的：小块负责精确检索，大块负责提供完整上下文。检索时先用小块锁定最相关的段落，再把关联的大块拉进来补充上下文，一并喂给 LLM。它的优点是既能精准定位，又不会丢失全局信息。代价是实现复杂，需要维护父子索引关系。适合的场景是复杂知识库——比如既有 FAQ 又有长篇技术文档，用户的问题可能只需要 FAQ 里的某个精确答案，也可能需要跨多个段落做综合推理。”

**我的回答**：

| 策略 | 核心思想 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **固定大小分块** | 按预设字符数或 Token 数直接切分 | 速度快、行为可预测 | 语义被截断，可能把一句话切到两个块里 | 日志文件、邮件、代码仓库、结构参差的语料库 |

| **基于段落分块** | 以完整段落为单位切分 | 每个分块对应一个独立子主题 | 段落长度差异大，长段落可能超限 | 长篇文档、研究论文、综述类文章 |

| **滑动窗口分块** | 创建相互重叠的窗口，相邻分块共享部分内容 | 上下文保持好，语义不中断 | 分块数量膨胀，存储检索成本上升 | 法律 RAG、金融分析、医学文献、跨句推理 |

**我的选型倾向**：对于文件上传场景，首期采用**固定大小分块 + 重叠**（如 chunk_size=512, overlap=50），实现简单、效果可预期；后续根据检索效果，考虑升级为**递归分割**或**语义分块**以获得更好的语义连贯性。

## 题12：解读 RAG 中的 Embedding 嵌入技术

**我的回答**：

Embedding（嵌入）是一种将高维、离散、非结构化数据映射到**低维连续向量空间**的技术。它解决了传统 one-hot 编码维度爆炸、语义丢失、无法计算相似度的核心痛点，具有低维稠密、保留语义、高效检索的特点。

在我项目里，我使用 bge-m3 模型将用户问题和知识库文档都转化为 1024 维向量，并存入 ChromaDB。当用户提问时，系统会将问题向量化，然后通过余弦相似度在 ChromaDB 中快速找到语义最接近的文档片段，最终将这些信息作为上下文注入 LLM，有效抑制了幻觉，提升了回答的专业性。

**项目关联**：
- 在 baby-ai 项目中，`get_embedding` 函数调用 bge-m3 模型，实时将文本转化为 1024 维语义向量。
- 核心实践：用户提问 -> Embedding -> ChromaDB 余弦相似度匹配 -> 返回最相关文档 -> 注入 LLM。选择 bge-m3 是因为其对中文友好、支持 8192 tokens 长文本，且同时具备 Dense 和 Sparse 检索能力，为后续升级混合检索留足了空间。

**关键词**：
Embedding（嵌入）、one-hot、余弦相似度、语义匹配、bge-m3、ChromaDB

---

**面试官可能追问**：

**Q1：为什么用余弦相似度而不是欧氏距离？**
🟢 当前状态：能讲清
- 在高维语义空间中，方向比绝对距离更能代表语义，余弦相似度只关注夹角，消除了文本长度对向量模的影响。
- 主流向量数据库（如 ChromaDB）对余弦相似度有底层优化，检索效率更高。
- 我可以举例：一句长回答和一句短总结，虽然向量长度不同，但语义相近时余弦值依然接近 1。

**Q2：bge-m3 的 1024 维是怎么定的？维度越高越好吗？**

🟢 当前状态：能讲清，有实测数据

> "不是越高越好。我用 3633 条数据做了维度对比实验，从 1024 维逐级降到 64 维，看 TOP-10 一致性：
>
> 1024 维：一致性 10/10，内存 28.4MB——精度完美。
> 512 维：一致性 6.9/10，内存 14.2MB——精度损失约 31%。
> 256 维：一致性 4.5/10——基本不可用。
>
> 所以 bge-m3 选 1024 维不是随便定的——这一档刚好保留了完整的语义信息，再降一档精度就明显受损。育儿场景下准确性是底线，内存减半换 31% 的精度损失不值得。但如果换到商品推荐场景，512 维的精度损失可能可接受，就能换来更低的存储成本。选维度本质上是对业务需求的权衡，不是越高越好，也不是越低越省。"

**Q3：bge-m3 和 text-embedding-3-small（OpenAI）你具体是怎么选的？**
🟢 当前状态：能讲清
- 从四个维度综合考量：
  1. 功能完整性：bge-m3 自带 Sparse（稀疏关键词） 检索，一个模型实现混合检索；OpenAI 仅 Dense。
  2. 成本与隐私：bge-m3 开源本地部署，零调用费、零数据出境风险；OpenAI 按 token 计费。
  3. 性能：本地无网络延迟，适合育儿问答的低延迟要求。
  4. 中文优化：bge-m3 在中文检索基准（如 T2Retrieval）上表现优秀。
- 结论：bge-m3 更适合我对功能、成本、隐私的综合要求。

**Q4：在你的项目里，如果不用 Embedding，会有什么后果？**
🟢 当前状态：能讲清
- RAG 管道基本失效。用户问“宝宝发烧”，知识库写的是“小儿发热护理”，传统关键词匹配会因用词不同直接漏掉正确答案。
- 只有 Embedding 能将“发烧”和“发热”映射到相近向量，实现语义理解，从而精准召回知识。
- 这体现了 Embedding 在非结构化文本检索中的不可替代性。


## 题13：RAG 中常用的 Embedding Model 嵌入模型

**我的回答**：

目前主流的中文 Embedding 模型有三类，我在项目里选的是 **bge-m3**，核心原因是它在功能完整性和中文检索上最匹配我的需求。

**先看对比**：

| 模型 | 维度 | 中文检索 | 功能特点 | 部署方式 | 问题 |
|------|------|:---:|------|---------|------|
| m3e-base | 768 | 中等 | 仅 Dense，512 tokens | 本地 | 轻量但功能单一，长文本需大量分块 |
| text-embedding-3-small | 1536 | 良好 | 仅 Dense | OpenAI API | 方便但收费，有网络延迟 |
| **bge-m3** | 1024 | **优秀** | Dense + Sparse + Multi-vector，8192 tokens | Ollama 本地 | ✅ 功能最全 |

**我的选型逻辑**：

**Q4：你提到 bge-m3 支持 Dense、Sparse、Multi-vector，能简单解释一下这三者的区别吗？**
🟢 当前状态：能讲清

> - **Dense（稠密向量检索）**：把整段文本压缩成一个固定维度的连续向量（如 1024 维），用余弦相似度做语义匹配。优点是能理解同义表达（“发烧”≈“发热”），缺点是可能忽略精确关键词。典型场景：语义搜索、模糊问答。
> - **Sparse（稀疏关键词检索）**：类似 BM25，把文本表示成高维稀疏向量，每个词有独立的权重分数。优点是精确匹配能力强，专有名词（如“对乙酰氨基酚”）不会漏；缺点是无法理解语义变体。典型场景：专有名词检索、精确匹配。
> - **Multi-vector（多向量检索）**：把文本中每个 token 都生成一个向量，用多个向量的组合来表示整段文本。优点是细粒度最高，能捕捉词与词之间的细微关系；缺点是计算量和存储成本显著增大。典型场景：需要精细语义匹配的专业检索。
>
> **我的使用策略**：目前主要用 Dense 做语义召回，后续会加上 Sparse 做精确匹配，两者组合成官方推荐的 Hybrid Retrieval。Multi-vector 暂时未落地，因为对硬件和索引结构要求较高，适合在知识库规模扩大、检索精度要求更高的阶段再引入。

## 题19：向量数据库的核心原理简述

**我的回答**：

向量数据库的核心原理是：**将非结构化数据通过 Embedding 模型转成向量，用近似最近邻算法做高效相似度检索。** 传统数据库做精确匹配，向量数据库做语义相似度匹配--把用户问题和知识库文档都转成向量，用余弦相似度或内积计算距离，找到最接近的 top-K 结果。

为了在海量向量中快速检索，向量数据库不会逐个比对（那样太慢），而是用 ANN 算法做近似匹配。常见的索引结构有 HNSW（分层可导航小世界图）、IVF（倒排索引）等，能在毫秒级返回结果。

我项目里用的 ChromaDB，就是基于 HNSW 索引的轻量级向量数据库。用户提问后，bge-m3 生成 1024 维向量，ChromaDB 用余弦相似度检索最相关的文档片段。

**项目关联**：
- baby-ai 使用 ChromaDB 存储向量，支持双集合并行检索（rag_samples + rag_medical）。
- 检索流程：用户提问 → bge-m3 向量化 → ChromaDB 余弦相似度搜索 → 返回 top-K → 注入 LLM。

**关键词**：向量数据库、近似最近邻算法、余弦相似度、HNSW、ChromaDB

## 题22：向量数据库中余弦相似度、欧几里得距离、曼哈顿距离的定义及差异

**我的回答**

1. 余弦相似度
**定义**：通过计算两个高维向量之间夹角的余弦值来衡量它们的相似度。
**核心特性**：取值范围在 [-1, 1] 之间，越接近 1 说明向量方向越一致，越相似。
**应用**：最适合做语义文本检索，因为它只关注向量的方向（语义内涵），不关注向量的长度（文本长度或词频大小）

2. 欧几里得距离

**定义**：计算两个向量空间点的真实直线距离（即多维空间中的勾股定理）
**核心特性**：数值越小，表示距离越近、越相似

**应用**：通常不用于语义文本检索，更常用于图片特征比对或数值型特征搜索

3. 曼哈顿距离

**定义**：两点之间只能沿着网格线（垂直或者水平）走的折线距离

**核心特性**：各维度坐标差值的绝对值之后

**应用**：超高维、稀疏数据或者计算速度要求极高、硬件资源有限的非语义特征检索场景。


## 题23：向量数据库的核心工作流程

**我的回答**

向量数据库的核心工作流程分四个阶段：

1. 向量化写入（入库）
文档经过解析、清洗、分块后，通过Embedding模型转为1024维向量，连同原始文本和元数据一起写入数据库，每条记录至少包含三部分：id（唯一标识）、vector（高维浮点数数组）、metadata（结构化标签）
2. 索引构建
入库后，向量数据维所有向量构建索引结构 --- 这里是它和传统数据库最大的工程差异，主流索引有两种：
  - HSNW(分层可导航小世界地图)：构建多层导航网络，上层是”高速公路“快速跨区域定位，下层是”街道“精确找到邻居。ChromaDB 用的就是这个，适合单机内存环境。
  - IVF（倒排文件索引）：先把向量聚类成若干簇，检索时只搜最近的几个簇，牺牲少量精度换更快的速度

3. 相似度搜索
用户提问 -》Query Rewrite -》Embedding 转为查询向量-》向量数据库用 ANN 算法在索引上快速定位近似区域，再用余弦相似度（或内积、欧氏距离）在候选集内精确排序，返回Top_k最相关的文档

之所以用ANN而非暴力搜索：百万级向量逐一比对需要几秒到几十秒，ANN通过HNSW等索引结构在毫秒完成，精确损失通常在5%以内。

4. 元数据过滤 + 结果返回
在向量检索的基础上叠加元数据过滤--比如用户问”宝宝发烧“，意图路由先识别为”常见病护理“类别，数据库只搜索category="常见病护理"的向量子集，大幅减少无关文档的干扰。最终返回TOP-K文档的原文和元数据，进入Rerank精排或直接注入Prompt

**项目关联**
baby-ai 使用 ChromaDB 存储向量，支持双集合并行检索（rag_samples + rag_medical），底层基于 HNSW 索引 + 余弦相似度计算。

关键词：向量数据库、HNSW、IVF、ANN、余弦相似度、元数据过滤、ChromaDB


## 题24：RAG 系统召回结果与用户 query 意图不匹配的改进方向

🟢 当前状态：已完成

**我的回答**

召回结果不匹配，可以从下面几个方向考虑：

1. **Query Rewrite(查询改写)*：处理用户输入中的代词、省略、模糊表达。比如用户在多轮对话中说”他应该吃什么药“，通过指代消解把”他“还原为”宝宝“，让检索更精确

2. **混合检索**：纯 Dense 对专有名词不敏感，纯 Sparse 无法理解语义变体。用 Dense + Sparse 混合检索，Dense 管语义泛化（”发烧“ ≈ ”发热“），Sparse 管精确匹配（对乙酰氨基酚），两者互补提升召回完整性。

3. **意图路由**：识别用户问题属于哪个类别，再根据类别优化后续的检索和生成策略，缩小检索范围，不同类别的问题可以选择合适的Prompt模版或者检索策略。

4. **Rerank 精排**：混合检索的排序靠的是 Bi-Encoder 的向量距离——query 和 doc 各自独立编码，从未在同一语境下深度交互，只能"隔空比手势"。Rerank 用 Cross-Encoder 把每对 (query, doc) 拼在一起做全 token 交叉注意力打分，"面对面审"，把真正匹配的文档提到 Top-3。我项目里用 BGE Reranker v2-m3，候选 100 条精排后只取 Top-5。

这四个方向叠加形成漏斗：Query Rewrite 让问题更完整 → 意图路由缩小范围 → 混合检索在范围内多路互补 → Rerank 精排优中选优。

**项目关联**：
- baby-ai 项目完整落地了上述四层方案：Query Rewrite（指代消解）、意图路由（6 类分类）、混合检索（Dense + Sparse）、Rerank 精排（BGE Reranker v2-m3）。

**关键词**：
Query Rewrite、意图路由、混合检索、Rerank、召回优化、漏斗式检索


## 题26：多知识库 RAG 中，查询效率与准确性的平衡及幻觉抑制方法

**我的回答**：

多知识库场景的核心挑战是：如果全库盲搜，检索效率低（数据量大、延迟高），且召回噪声多（不相关类别的文档混入），LLM 容易被无关信息误导产生幻觉。

我的平衡策略分两层：
1. **意图路由缩小范围**：先用关键词匹配识别用户问题类别（常见病护理/喂养/睡眠/生长发育/疫苗与体检/日常护理），定向检索对应类别文档，减少无关数据干扰。
2. **混合检索多路互补**：在定向检索的基础上，Dense 管语义泛化，Sparse 管精确关键词，两者融合确保在缩小后的范围内不遗漏关键信息。

此外，我的 Prompt 中有条件检查——要求 LLM 先判断检索文档是否包含所需信息，不包含则诚实说明，而不是强行编造，这是第二层幻觉抑制。

**项目关联**：我的知识库有 rag_samples（AI 生成的高质量数据）和 rag_medical（真实医疗对话数据）两个集合。意图路由识别类别后，可以按类别优先检索对应集合，减少跨库盲搜的噪声。

## 题33：RAG 中向量数据库的必要性及与传统数据库的差异

**我的回答**

**必要性：**

1. 解决传统数据库无法做语义相似度检索的缺点，即同样的意思不同的表达，传统数据库无法将这两个词匹配到一起。

2. 优化高维向量的近似最近邻搜索，向量数据库专为高维空间设计，通过构建 HNSW（层次化导航小世界图） 或 IVF（倒排文件索引） 等高级索引结构，将计算复杂度从 O(N) 降至 \(O(\log N)\)，在百万至亿级数据量下实现毫秒级的 近似最近邻搜索（ANN）。

3. 实现多路混合检索，现代化向量数据库（如 Milvus、Qdrant）原生支持在一个表里同时存储、索引和查询这两种向量，并自带重排合并机制，大大简化了 RAG 的后端架构。


**差异**

| 对比维度 | 传统数据库（如 MySQL、PostgreSQL） | 向量数据库（如 Milvus、Pinecone、Chroma） |
|---|---|---|
| 核心数据类型 | 结构化文本、数字、布尔值（表格形式） | 高维浮点数数组（向量 / Embedding） |
| 检索核心逻辑 | 精确匹配（等于、大于、LIKE 模糊匹配） | 近似匹配（计算内积、余弦相似度、欧氏距离） |
| 查询结果 | 非黑即白（要么符合条件，要么不符合） | 概率排序（按相似度得分从高到低返回 Top-N 结果） |
| 底层的索引结构 | B+ 树、哈希表（为了快速定位到具体的某一行） | 空间图（HNSW）、空间树、倒排（IVF）（为了在多维空间中拉网快速找邻居） |
| 修改数据的代价 | 极低。改一个字或一条数据，几乎不影响整体索引 | 较高。大量插入新向量时，底层空间图需要重新计算、连线和平衡 |
| 应用场景 | 订单系统、用户账户、财务报表（不容许任何误差） | RAG 知识库、图片/视频特征检索、AI 推荐系统 |


## 题34：RAG 系统利用元数据过滤提升检索精度的方法

**我的回答**：

元数据过滤是在向量检索的基础上，利用文档的结构化标签（如来源、类别、日期、评分等）做前置或后置过滤，缩小候选范围，提升检索精度。

**我的实现方式**：
1. 灌库时为每条文档写入元数据——category（分类标签）、source（数据来源）、score（质量评分）
2. 意图路由先识别用户问题类别（如“常见病护理”），得到 category 标签

4. 在过滤后的候选集内再做混合检索（Dense + Sparse）

这是“两层漏斗”：第一层元数据过滤粗筛（快速排除大量不相关文档），第二层混合检索精排（语义+关键词互补）。相比直接全库混合检索，元数据过滤能显著减少噪声文档的混入。

**还可以扩展的元数据维度**：来源过滤（只检索权威指南，排除用户上传文档）、时间过滤（只检索最新版本的知识条目）、评分过滤（只检索高质量 Q&A）。

**关键词**：元数据过滤、意图路由、混合检索、两层漏斗

## 题36：向量索引的构建、使用方法及 HNSW 与 IVF 的差异

> ⚠️ 疲劳状态下完成，面试前需重新过一遍

**我的回答**

**向量索引的构建**：

向量存入数据库后，如果不建索引，每次检索都要把查询向量和全库所有向量逐一算距离（暴力搜索），百万级数据延迟不可接受。索引的作用就是预先为向量建立"导航结构"。

以 HNSW 为例：入库时逐条插入向量，按概率随机分配到不同层级——顶层节点少（高速公路），底层节点多（街道）。每层为向量建立图连接，连接到该层已有的最近邻居。插入完成后，HNSW 的多层导航图就构建好了。

**向量索引的使用方法**：

检索时分两步：先在顶层快速跨区域定位，找到大致区域后逐层下沉，到达底层后在候选邻居中精确计算余弦相似度排序，返回 Top-K。整个过程只需计算几十到几百次距离，而非暴力搜索的百万次。

**HNSW 与 IVF 的核心差异**：

| 维度 | HNSW | IVF |
|------|------|-----|
| 原理 | 图索引：构建多层导航网络，检索时逐层下沉 | 聚类索引：先把向量分成若干簇，检索时搜最近的簇 |
| 检索方式 | 沿图连线贪心搜索，在层与层之间跳跃定位 | 查询向量先和簇中心比距离，选最近的 N 个簇，在簇内暴力搜索 |
| 索引构建 | 逐条插入，动态构建图连接 | 需要先对所有向量做 K-Means 聚类训练 |
| 检索速度 | 极快，适合内存内场景 | 依赖簇的数量和簇内数据量 |
| 内存占用 | 较高（需要维护图连接） | 较低（只存簇中心 + 簇内向量） |
| 适用场景 | 单机、中小规模、低延迟 | 分布式、超大规模、可接受离线训练 |

**ChromaDB 为什么选 HNSW**：Chroma 定位是嵌入式轻量数据库，通常在单机内存环境下运行。HNSW 在内存内的检索延迟极低，且不需要 IVF 那样的离线聚类训练步骤，和 Chroma"开箱即用"的体验一致。如果数据上亿级、需要分布式部署，Milvus 的 IVF 方案更合适。

**关键词**：向量索引、HNSW、IVF、ANN、聚类、图索引、ChromaDB

## 题39：RAG 中 Embedding 向量化的定义及工作原理

**我的回答**

**定义**：
Embedding嵌入是一个将高维、离散、非结构化的数据映射到低维连续向量空间的技术，输出的向量是一个固定维度的浮点数数组（如1024维），语义相近的文本在向量空间中距离更近。

**工作原理**：
Embedding 模型的核心是Transformer 架构，以bge-m3为例，工作流程如下：
  1. 分词：输入文本被切分维token序列，每个token映射为一个初始向量
  2. 深层编码：经过多层Transformer的自注意力计算，每个token的向量不断吸收上下文信息，最终变成“理解了整句话含义”的表示。这一步是动态的 -- 同一个词在不同上下文中的最终向量不同。
  3. 池化：将整段文本的所有token向量聚合为一个固定维度的句子级向量。bge-m3取token的最终隐藏状态为整段文本的语义表示，输出1024维的稠密向量。

**项目关联：**
baby-ai中 get_embedding 函数调用bge-m3模型，用户问题和知识库文档实时转化为1024维向量，存入ChromaDB。检索时用余弦相似度找语义最接近的文档片段。

**关键词**
Embedding、向量化、分词（Tokenization）、Self-Attention、池化（Pooling）、[CLS] token、bge-m3、1024维、one-hot、余弦相似度、ChromaDB

## 题42：RAG 检索中 Top-K 的含义及 K 值的确定方法

> ⚠️ 疲劳状态下完成，面试前需重新过一遍

**我的回答**

**定义**：Top-K 是指检索阶段返回的最相关文档数量。检索系统从知识库中找到与查询最相似的 K 个文档片段，作为 LLM 生成的参考资料。

**K 值不是越大越好，也不是越小越好**：

- **K 太小**（如 Top-3）：可能漏掉关键文档。用户问"对乙酰氨基酚怎么吃"，如果第 1、2 名是护理指南，第 4 名才是剂量表，K=3 就会漏掉最关键的剂量信息。
- **K 太大**（如 Top-50）：噪声文档混入，LLM 被无关信息干扰，容易产生幻觉。同时 Token 消耗线性增长——每多一篇文档就多几百字注入 Prompt。

**我项目的做法——两层漏斗**：

| 阶段 | K 值 | 作用 |
|------|:---:|------|
| 粗筛（混合检索） | Top-100 | 保证召回覆盖率，"宁可多捞，不能漏掉" |
| 精排（Rerank） | Top-5 | 从 100 条中精筛出最相关的，注入 LLM |

粗筛取 100 条是为了给 Rerank 留足够的候选池——如果粗筛只取 10 条，某些"语义一般但关键词精确匹配"的文档根本进不了候选池，Rerank 再强也无从筛起。Rerank 后只取 5 条注入 Prompt，是为了保证每一条都是高质量参考资料，同时控制 Token 成本。

**确定 K 值的思路**：

1. **看场景对漏召的容忍度**：医疗问答漏不得，K 偏大；闲聊式问答可以偏小。
2. **看 LLM 上下文窗口**：K 条文档的总 token 量不能超出窗口限制。
3. **通过实验验证**：用不同 K 值（10/20/50/100）跑同一批测试问题，统计检索命中率的边际收益。K 增加不再带来命中率明显提升时，就是最优值。

**项目关联**：baby-ai 中混合检索取 top-100 候选，Rerank 后取 top-5 注入 Prompt。

**关键词**：Top-K、检索精度、召回率、Rerank、两层漏斗、Token 成本

| **省略补全** | “有什么副作用吗”（上一轮问了“对乙酰氨基酚”） | “对乙酰氨基酚有什么副作用吗” | 省略主语导致检索模糊 |
| **查询扩展** | “宝宝发烧” | “小儿发热 发烧 体温升高 退烧” | 增加同义词覆盖更多相关文档 |
| **模糊修正** | “宝宝晚上总是醒，睡不好” | “宝宝频繁夜醒 睡眠障碍 夜惊” | 口语化表达转为更精确的术语 |

**我的实现方式**：

三步走策略：
1. **检测是否需要改写**：用正则匹配用户输入中的代词（他/她/它/这个/那些等），如果检测到代词且有对话历史，触发改写；否则跳过，直接使用原始查询。
2. **调用 LLM 做指代消解**：将当前查询和最近几轮对话历史拼接成 Prompt，让 DeepSeek 把代词还原为具体指代对象。
3. **降级兜底**：如果 LLM 调用失败（超时/异常），直接退回原始查询，不阻塞检索主流程。

**效果**：不改写时，纯 Dense 检索可能勉强召回相关内容（语义相近），但 Sparse 检索完全失效（“他”无法做关键词匹配）。改写后，Dense 和 Sparse 都能精准工作，召回完整性显著提升。

**项目关联**：
- 代码位置：`app/services/query_rewrite.py`
- 集成点：`chat_service.py` 的 `get_rag_context` 中，检索之前调用
- 依赖对话历史管理（`session_manager.py`）提供上下文

**关键词**：Query Rewriting、指代消解、查询扩展、省略补全、降级策略

## 题51：RAG 中文档切割中 chunk_size 与 overlap 的设置方法

**我的回答**

chunk_size：
- 控制每个分块包含多少内容（按字符数或者token数）
- 设太大，单个分块包含多个主题，检索时向量相似度匹配不够精准。
- 设太小，语义碎片化，一个完整的知识点被切到两个分块里，两个块的向量都不足以回答用户问题，双双排不上Top_k
- 项目考量：bge-m3 支持8192 tokens输入，但实际设 512 tokens左右，因为Q&A数据集中每条问题本身就是完整的一个语义单元，不需要额外切分。

overlap：
- 相邻分块之间共享一段内容，防止关键信息恰好落在分块边界被切断
- 设太小或不设：比如”对乙酰氨基酚“在块A末尾，”剂量“在块B开头，两个块单独看都不完整，检索时可能都排不上号
- 设太大：冗余增加，同一段路内容在多个分块中重复出现，浪费存储和Token
- 一般经验：overlap 去chunk_size的10%-20%。

**确定最优值的思路：**
先用经验值（如512/50）跑通链路，再用评估脚本对比 256/512/1024 三种chunk_size 的检索命中率差异，选综合表现最好的。

**项目关联**
baby-ai 当前知识库是预处理好的结构化 Q&A 数据集，每条数据本身就是一个完整分块。文件上传功能上线后，对用户上传的 PDF/Word 等长文档，首期用固定大小分块（chunk_size=512、overlap=50），后续根据评估结果调参。

**关键词**

chunk_size、overlap、分块策略、固定大小分块、检索精度、上下文完整性


## 题52：混合检索的定义及向量检索与关键词检索的结合实现

**我的回答**

**定义**：
混合检索是将 Dense（稠密向量检索，管语义）和 Sparse（稀疏关键词检索，管精确）结合到一起的检索方案。它解决了纯 Dense 对专有名词不敏感、纯 Sparse 无法理解同义表达的问题，两者互补，显著提升召回完整性。

**实现方式**：
目前主流的融合策略有两种，我在项目里都落地了：

**1. 线性加权**
核心思想是给 Dense 和 Sparse 各一个权重，加权求和：`最终分数 = α × Dense分数 + (1-α) × Sparse分数`。
- **融合步骤**：① 两路各取 top-K 候选 → ② Min-Max 归一化把两路分数统一到 0~1 → ③ 加权求和 → ④ 按最终分数重新排序。
- **α 的作用**：α 越接近 1 越偏重语义，越接近 0 越偏重关键词。经 24 条评估数据对比，α=0.5 忠实度最优（4.29），说明育儿场景下语义和关键词同等重要。原设 0.7 已修正为 0.5。
- **关键坑点**：必须归一化！Dense 的余弦相似度是 0~1，Sparse 的 BM25 分数无上限，不归一化会导致 Sparse 分数碾压 Dense，α 形同虚设。

**2. RRF（倒数排名融合）**
核心思想是不看具体分数，只看排名位置：`RRF分数 = 1/(60 + Dense排名) + 1/(60 + Sparse排名)`。
- **为什么要用 RRF**：线性加权的最大问题是两路分数量纲不同，必须归一化才能相加。RRF 只关心排名，天然绕过了量纲问题，也不需要手动调 α。
- **效果**：如果一篇文档 Dense 排名第 1 但 Sparse 排名第 20，线性加权可能仍把它排得很高（Dense 分数高），而 RRF 会因为 Sparse 排名差而压低它。RRF 奖励的是“两路都排得靠前”的文档，对噪声更鲁棒。

**项目关联**：
在我的育儿助手项目里，我用 bge-m3 一个模型同时输出 Dense 和 Sparse 两路向量，省去了额外部署 BM25 引擎的成本。融合策略上，我先用线性加权（α=0.7）快速验证混合检索的可行性，后续切换到 RRF 以获得更鲁棒的融合效果，无需手动调参。实测中，对于“宝宝发烧对乙酰氨基酚怎么吃”这样的复合查询，混合检索能将语义相关的护理指南和精确匹配的剂量表同时召回，显著提升了检索命中率。

**关键词**：
混合检索、Hybrid Retrieval、Dense、Sparse、线性加权、归一化、RRF、bge-m3

---

**面试官可能追问**

**Q1：bge-m3 输出 Sparse 向量是怎么做到的？和传统 BM25 有什么区别？**

>"bge-m3 生成 Dense 向量的同时，会对每个 token 输出一个权重分数，这些权重组合起来就是 Sparse 向量——本质上和 BM25 一样，都是给每个词算重要性权重。区别在于：传统 BM25 需要单独部署一个检索引擎，用 TF-IDF 公式手动计算词权；bge-m3 的 Sparse 权重是模型自己学出来的，不需要额外部署，且和 Dense 共享同一个编码过程，几乎零额外成本。这让我一个模型就能实现混合检索，架构简洁很多。"


*Q2：你的混合检索中，Dense 和 Sparse 各取多少条候选？为什么这样设计？**
🟡 当前状态：能讲清设计逻辑，待做定量对比 → 📅 阶段 2.7（对比不同候选数量的检索命中率）

> “目前两路各取 20 条候选，然后合并后筛选 top-5 返回。取 20 条是为了给融合阶段留足够的候选池——如果某一路只取 5 条，另一路的重要文档可能根本没进候选池，融合就无从谈起。具体数字是经验值，如果要精确确定，我会用不同候选数量（10/20/30）做对比实验，看检索命中率的边际收益。超过一定数量后，新增的候选大多是噪声，反而拖慢检索速度。”


**Q3：如果某篇文档在 Dense 里排前 3，但在 Sparse 里完全没匹配（分数为 0），你会怎么处理？**
🟢 当前状态：能讲清

> “这种情况说明这篇文档语义高度相关但不包含查询中的精确关键词。用 RRF 时，我会给缺失的那一路分配一个较大的排名值（如候选池大小 + 1），这样文档的 RRF 总分会被适当压低，但不会被完全排除。用线性加权时，缺失那一路的分数为 0，最终分数完全由另一路决定。两种方式都不会直接丢弃它，只是降权。如果业务上认为精确关键词必须命中，可以设一个 Sparse 分数的最低阈值，低于阈值就直接过滤——但这需要根据具体场景权衡。”

🟡 当前状态：概念可讲，定量数据待测 → 📅 阶段 4.5（模型部署方案对比 + 性能压测）

> “Sparse 检索（BM25）的计算复杂度远低于 Dense 的向量相似度计算，所以额外延迟主要来自分词和分数融合，实际增量很小——在我的本地环境下，从纯 Dense 切换到混合检索，响应延迟几乎无感知变化。bge-m3 的一个优势是 Sparse 权重和 Dense 向量在同一次编码中产出，不需要额外调模型，比单独部署 BM25 引擎的方案快得多。如果要定量验证，我会用压测工具对比两种方案的 P50/P99 延迟。”

**Q5：你提到了 Min-Max 归一化，还有没有其他归一化方法？为什么选 Min-Max？**
🟡 当前状态：能讲清概念 → 📅 阶段 2.7（评估搭建时可顺带验证不同归一化方法的效果）

> “常见的还有 Z-score 标准化（减均值除标准差），它适合数据服从正态分布的场景。混合检索中的 Sparse 分数通常是长尾分布——少数文档分数很高，大部分集中在低分段。Min-Max 更适合这种分布，因为它直接映射到 0~1，不受分布形态影响。另外 Min-Max 保留了原始分数的相对大小关系，这正是线性加权需要的信息。如果用 RRF，归一化就完全不需要了。”


## 题50：RAG 系统效果评估方法及检索、生成环节的核心评估指标

🔴 待整理 → 📅 与题9一起完成
- 核心内容：检索环节看 Context Recall/Precision，生成环节看 Faithfulness/Answer Relevancy

**我的回答**

RAG 系统的效果评估需要从检索和生成两个环节分别衡量，核心是三个指标：

检索环节 — Context Recall（上下文召回率）：
衡量"该召回的是否都召回了"。这是 RAG 链条的第一关——如果正确答案根本没进候选池，后面生成再强也是无米之炊。这个指标帮你回答"问题出在检索还是生成"。

生成环节 — Faithfulness（忠实度）和 Answer Relevancy（答案相关性）：

忠实度衡量回答是否基于检索文档，有没有编造。育儿场景下这是生命线——幻觉可能误导用药。
答案相关性衡量回答是否贴合用户问题，防止模型拿到正确资料却东拉西扯。
为什么优先选这三个而非其他：
核心考量是实施成本和诊断效率。忠实度和答案相关性不需要 ground truth，RAGAS 内部用 LLM 自动评判；只有上下文召回率需要轻量标注（标注"应该包含哪些信息点"，不必写完整标准答案）。三者结合能在 20-30 条数据、每条 30 秒标注的成本下跑通完整闭环。而 Answer Correctness 需要逐条标注完整标准答案，成本高出 5-10 倍，适合基线达标后再引入。

三个指标形成诊断链：
召回率低 → 问题在检索，查 Embedding/分块/混合策略；
召回率高但忠实度低 → LLM 在编造，查 Prompt 和 Rerank；
两者都高但答案相关性低 → LLM 在跑题，查 Prompt 约束和结构化输出。

**项目关联：**
baby-ai 基于 RAGAS 搭建评估脚本，24 条测试数据覆盖 6 个育儿类别，已跑通 α 值对比和 RRF vs 线性加权实验，通过评估数据将默认 α 从 0.7 修正为 0.5。

**关键词：**
RAGAS、Context Recall、Faithfulness、Answer Relevancy、诊断链、评估基线
