## 题5：MCP 协议的定义及在大模型系统中的作用

**我的回答**

**定义**：MCP（Model Context Protocol）是一个开放、标准化的通信协议，定义 AI 应用（客户端）与外部工具/数据源（服务器）之间如何"发现能力"、"调用功能"和"传递上下文"的通用交互规范。

**本质**：一套通信规则。目标是打破 AI 应用与工具之间的私有壁垒——M×N 的集成复杂度变为 M+N，一次编写工具、任意 AI 应用调用。

**在大模型系统中的核心作用**：

1. **标准化工具集成**：MCP 出现之前，每个 AI 应用要为每个外部工具编写定制集成代码。MCP 通过统一协议，开发者只需为工具实现一次 MCP 接口，就能被所有支持 MCP 的 AI 应用使用。

2. **解耦模型与工具**：MCP 不依赖特定 LLM 能力——即使模型不支持 Function Calling，也能通过 Prompt 解析 + MCP 传输实现工具调用（纯 MCP 模式）。

3. **动态工具发现**：客户端启动时通过 `tools/list` 自动获取服务器支持的工具清单，新增工具无需改 Agent 代码或重启服务。

4. **上下文传递**：MCP 支持在请求中传递上下文信息（用户 id、对话历史），保持多轮交互的完整性。

**关键词**：MCP、模型上下文协议、标准化、工具集成、动态发现、M×N→M+N

## 题6：MCP 架构的核心组件

**我的回答**

MCP 采用客户端-服务器（Client-Server）架构，三个核心角色：

| 角色 | 职责 | 举例 |
|------|------|------|
| **MCP Host** | AI 应用本身，发起连接的一方 | Claude Desktop、IDE 插件、自建 Agent |
| **MCP Client** | 运行在 Host 内部，与单个 Server 建立一对一连接 | 协议客户端库（如 `mcp` Python 包） |
| **MCP Server** | 轻量级服务程序，通过 MCP 暴露功能 | 天气 API Server、文件系统 Server |

**Server 端三大核心原语（Primitives）**：

| 原语 | 类型 | 作用 | 示例 |
|------|:---:|------|------|
| **Tools** | 可执行 | LLM 可通过 FC 触发的有副作用操作 | `get_weather(city)`、`send_email()` |
| **Resources** | 只读 | 由 URI 标识的静态/动态数据，作上下文读取 | `file:///logs/today.txt` |
| **Prompts** | 模板 | Server 预定义的带参数工作流模板 | `review-code` 代码审查模板 |

**关键词**：MCP Host、MCP Client、MCP Server、Tools、Resources、Prompts

## 题7：MCP 协议支持的两种模式

**我的回答**

MCP 支持两种传输模式，按部署场景选择：

| 模式 | 全称 | 通信方式 | 适用场景 |
|:---:|------|------|------|
| **Stdio** | 标准输入输出 | 进程间标准 I/O 管道 | MCP Server 和 AI 应用在同一台机器上，延迟极低 |
| **SSE** | Server-Sent Events (HTTP) | HTTP 长连接推送 | Server 部署在云端，远程调用 |

底层统一采用 **JSON-RPC 2.0** 规范进行异步双向请求与响应。传输层的选择对上层是透明的——无论用 Stdio 还是 SSE，工具调用的消息格式相同。

**我的项目考量**：baby-ai 是本地部署的单机应用，MCP Server 和 FastAPI 在同一台机器上，用 Stdio 模式即可，无需引入 HTTP 开销。

**关键词**：Stdio、SSE、JSON-RPC 2.0、传输层

## 题8：MCP 与 Function Calling 的核心差异

**我的回答**

| 对比维度 | Function Calling | MCP |
|------|------|------|
| 本质定义 | 技术能力：LLM 输出结构化 JSON 指令 | 通信协议：AI 应用与工具交互的通用规则 |
| 核心解决 | 模型输出格式问题 | 系统集成标准化问题 |
| 标准化程度 | 厂商私有，各模型格式不同 | 跨模型跨平台通用 |
| 角色定位 | "翻译官"：自然语言→函数名+参数 | "通用插座"：AI 应用↔工具服务器 |
| 内容范围 | 仅针对 Action（工具调用） | Tools + Resources + Prompts |
| 交互模式 | 点对点单次请求-响应 | C/S 架构：能力协商+动态发现 |
| 依赖关系 | 需模型原生支持 tool_calls | 不依赖模型能力 |

**关系**：不是替代，是"决策层"与"传输层"的垂直协作——FC 做决策（点哪道菜），MCP 做路由（把菜单传给后厨）。

**关键词**：Function Calling、MCP、决策层、传输层、标准化

## 题9：MCP 协议的核心工作流程

**我的回答**

MCP 的完整流程分五步：

1. **连接与协商**：Client 通过 Stdio/SSE 连接 Server，双方声明各自支持的功能（如支持的协议版本、传输方式）。

2. **工具发现**：Client 发送 `ListToolsRequest`，Server 返回当前可用的所有工具清单（含 name/description/inputSchema）。

3. **格式转换**：Agent 将 MCP 工具清单格式转换为 LLM 要求的 Function Calling Schema（如 DeepSeek 的 `tools` 参数格式）。

4. **FC 决策**：LLM 根据用户意图，从工具清单中选出目标函数并输出 tool_calls JSON。

5. **MCP 执行**：Agent 将 tool_calls 封装为 MCP 标准的 `CallToolRequest`（JSON-RPC 2.0 格式），发送给 Server 执行，Server 返回结果后 Agent 将 Observation 回填给 LLM。

**关键词**：能力协商、动态发现、ListToolsRequest、CallToolRequest、JSON-RPC
