## Rerank 概念 和 原理

**概念：**

Rerank 模型是 Cross-Encoder 架构，把 query 和 doc 拼接后做全文本交叉注意力，输出精确的相关性分数。和 Bi-Encoder（独立编码再余弦比较）相比，精度更高但计算更慢，只能用于候选集精排。


### 为什么在混合检索后必须进行 Rerank（重排）？

在 RAG 应用中，即使配置了最完美的双路检索（HNSW 稠密向量 + 倒排稀疏索引），捞上来 Top-100文档依然存在两个致命的痛点，这就是为什么要引入 Rerank 进行二次精筛。

1. 粗筛模型的智商局限
    无论是BGE-M3 Dense 或者 Sparse 检索，它们底层都属于双塔架构（BI-Encoder）。即文档向量和 查询向量是独立计算出来的，最后只在数据库做一次简单的空间距离或内积比对比。它们没有让用户问题和文档正文发生真正的”字对字“ 深度上下问交叉理解。

2. RRF算法的粗暴组合
    RRF算法纯粹是根据排名进行数学求和，完全没考虑文档相关性的绝对分值差异。例如：某篇极其精准的文档和一篇勉强相关的文档，在融合时可能只因为名次差了一两位，就被抹平了差距。

- 粗筛（向量/传统检索）： 像保安大爷，要在 100 万人里，通过看一眼长相（语义），5 毫秒内帮你快速圈出最可疑的 100 个人。

- 精筛（Rerank）：像审讯专家，不再面对 100 万人，只对保安大爷圈出的这 100 个人进行逐一深度审讯（全文本交叉比对），重新打分，挑出真正掌握核心线索的 Top-3 喂给大模型。

**我的理解**：

粗筛（Bi-Encoder）是"隔空比手势"——问题和文档各自独立编码，从未在同一语境下交互过，只能通过向量距离猜相关性。召回的是"可能相关"。

精筛（Cross-Encoder）是"面对面审"——把问题和文档拼在一起做全 token 交叉注意力，真正理解两者的匹配程度。选出的是"确实相关"。

### 深度选型对比与选型原则

1. 开源自检内部较量：小模型（M3）vs 大模型（Gemma）

| 模型大小 | 计算开销 | 智商差异 |
|---------|----------|---------|
| 小模型 | bge-reranker-v2-m3只有几百M到1G多大小，甚至可以在普通的单张 T4 或 4090 显卡上实现高并发、毫秒级的重排响应 | 不如大模型
| 大模型 | bge-reranker-v2-gemma 动辄 2B（20亿参数）到 7B。它需要消耗数 GB 到十几 GB 的显存。如果粗筛丢给它 50 篇文档，它的全注意力机制（Full Attention）会导致严重的排队延迟。| 智商极高

2. 商业API的对决：Cohere（表格专家）vs Jina（长文本专家）

- Cohere v3 的降维打击：传统的重排模型只能吞纯文本（Strings）。如果你的 RAG 系统清洗（Cleaning）出来的文档包含大量的结构化 JSON 或者是带有行列逻辑的 Markdown 表格，传统的 Reranker 在做全注意力计算时，行列关系会错乱。Cohere v3 经过专门的结构化微调，它能“看懂”表格里哪一行对应哪一列，打分精准度极高。

- Jina v2 的空间魔力：大部分 Reranker 只要文本超过 4000 字（Tokens）就会直接截断（Truncation），后面的内容根本不参与重排。Jina 原生支持 256k 的超长窗口。如果你的 RAG 预处理阶段分块（Chunk Size）设得很大（比如单块 2000 字），为了防止被模型截断，Jina 是唯一的完美选择。


### 最佳实践

1. 策略1：数据不出网 + 追求性价比
    1. 首先组合：粗筛使用 BGE-M3 ，重排使用本地部署的 bge-reranker-v2-m3 。
    2. 理由：两个模型同根同源，数据分布天然契合。在普通轻量级GPU服务器上即可跑通，单次重排延迟控制在50ms以内，性能极其均衡。

2. 策略2：面向极端逻辑问题 + 内部政企场景
    1. 首选组合：本地部署 bge-reranker-v2-gemma-v2（配合FlashAttention加速）
    2. 理由：牺牲一部分并发速度，换取极高的逻辑对齐准确率，确保给大模型的上下文100%正确。

3. 策略3： 面向出海、电商或高价值商业SAAS

    1. 首先组合：直接调用 Cohere Rerank v3 API
    2. 理由：无需维护任何GPU推理服务器，且表格、多语言、用户意图重排能力属于工业级天花板级别。


## Rerank 的异常处理策略

在 RAG 工业界落地中，“粗筛出的文档总长度超过 Rerank 模型的最大上下文限制（如 4096 或 8192 Tokens）” 是一个非常经典的工程痛点 [1, 2]。如果直接把超长文本塞进去，轻则导致模型在底层硬生生截断（Truncation）、漏掉后半段的黄金答案；重则直接引发 GPU 显存爆炸（OOM, Out of Memory）。

解决方案分为：应用层策略优化（做减法、截断）、模型/底层硬核替换（做加法、扩容）

### 应用层策略优化

1. 严格控制粗筛召回的文档数量（Top-K 漏斗控制）

粗筛阶段限制去取Top-K 30 到 Top-k 50。

2. 对每个文本块执行动态 Token 计算与单个截断
- 做法：在将Query与每一篇Doc拼成Pair时，先在代码里用分词器计算该Doc的Token数

- 工程标准：如果某个chunk异常的长，在代码中通过tokens[:max_doc_tokens] 进行前置主动截断。将剩余的空间留给其他更多的候选文档。

3. 动态滑动窗口切分重排

- 做法：如果粗筛捞上来的文档过长，又不能截断，把这篇文章在内存中用滑动窗口再次切碎（例如每512字一段，重叠50字），然后把这几段分别与Query拼接组成多组Pair丢给Reranker。

### 模型与底层硬核替换

1. 切换为 超长上下文重排模型

2. 底层开启硬件级注意力加速技术


面对 Rerank 输入过长，在你的 RAG 架构设计中应该遵循这套流水线标准：

1. 控制源头：清洗和分块阶段，单块大小不要设置的太大，尽量控制在512-1000tokens

2. 设置关卡：混合检索时，只放 30 - 50篇 文档进入重排候选集。

3. 代码兜底：在调用Reranker之前，使用分词器开启 truncation=True ，强制限制单次拼接的最大长度，确保显存绝对安全。

4. 业务升级：如果发现截断后总是漏掉重要内容，立刻将重排模型无缝平替为支持256K的Jina Reranker V2。


## Rerank 在我的 RAG 管道中的位置

用户问题
  → Query Rewrite（指代消解）
  → 意图路由（6类分类）
  → 混合检索（Dense + Sparse，粗筛候选 100 条）
  → **Rerank 精排（BGE Reranker v2-m3，选出 top-5）**
  → Prompt 拼接
  → LLM 生成

粗筛阶段的 100 条候选文档经过 Rerank 后，只有最相关的 5 条进入 LLM 的上下文窗口。
这确保了 LLM 收到的每一条参考信息都是高质量的，从源头减少了幻觉。

## 我的项目实践

混合检索（Dense + Sparse）-》候选集（100条）-》Reranker 精排 -》Top-5 -》注入LLM

- 选型：BGE Reranker V2 M3（本地服务， FastAPI部署）
- 备选：Deepseek排序（调用LLM对文档打分）
- 兜底：原始向量检索结果（取Top-k直接使用）
- 这套三级降级保证了任何异常下系统都不会崩溃。


bge-reranker-v2-m3：

输入格式：[CLS] Query 文本 [SEP] Document 文本 [SEP]， 数据结构大概为在 Python 的 transformers 库或官方 FlagEmbedding 库中，输入格式非常直观，是一个 嵌套列表（二维数组），每个子列表刚好包含两个字符串：[Query, Document]。类似于这种

```
# 把用户问题直接和每个文档中的每个文本进行拼接即可。
pairs = [[req.query, doc.text] for doc in req.documents]
```

输出分数： 
- 未经归一化的原始对数几率（Raw Logits），没有固定的上限和下限。分数越高代表越相关。
- 可以通过激活函数转化为 概率得分（Probability Score）。严格在0~1之间，代表文档能够回答该问题的绝对相关性概率。
