## 为什么需要专门评估 RAG？

RAG 不同于 传统NLP任务（如分类、翻译），它的输出是非确定性的，无法简单用 BLEU、ROUGE等指标衡量。

### 为什么 BLEU/ROUGE 在 RAG 场景下彻底失效？

1. **参考标准缺失**： 翻译有“标准译文”，摘要有“标准摘要”。但 RAG 的输入是**开放域问题**（如昨天发生某件突发事件），输出基于**实时检索到的私有文档**，。这个“标准答案”本身就是动态生成的，无法预先人工标注。

2. **语义平权问题**：如果正确答案是“苹果公司”，模型输出“iPhone制造商”或“蒂姆库克所在企业”。BLEU算字面重叠会得零分，但在RAG语境下这些答案完全正确。

3. **非确定性输出**：RAG 输出高度开发、非确定性。

### 1. BLEU（Bilingual Evaluation Understudy）

| 属性 | 说明 |
| :--- | :--- |
| **诞生背景** | 机器翻译任务 |
| **核心逻辑** | 基于 **精确率（Precision）**，计算生成文本中的 N-gram 在参考文本中出现的比例 |
| **特色机制** | **短句惩罚（Brevity Penalty）**：防止生成过短文本得高分 |
| **适用场景** | 机器翻译、标准答案相对固定的封闭式任务 |
| **局限性** | 只看字面重叠，不关注语义；对同义词、语序变换敏感 |

### 2. ROUGE（Recall-Oriented Understudy for Gisting Evaluation）

| 属性 | 说明 |
| :--- | :--- |
| **诞生背景** | 文本摘要任务 |
| **核心逻辑** | 基于 **召回率（Recall）**，计算参考文本中的 N-gram 被生成文本覆盖的比例 |
| **常用变体** | ROUGE-N（N-gram 匹配）、ROUGE-L（最长公共子序列 LCS） |
| **适用场景** | 文本摘要、要点覆盖类任务 |
| **局限性** | 同样无法理解语义，只做机械字面匹配 |

### 3. BLEU vs ROUGE 核心对比

| 维度 | BLEU | ROUGE |
| :--- | :--- | :--- |
| **侧重点** | 精确率（Precision） | 召回率（Recall） |
| **关心问题** | 生成的内容是否精准？ | 参考内容是否被完整覆盖？ |
| **惩罚对象** | 生成文本过短 | 遗漏参考文本中的要点 |
| **典型场景** | 翻译要精准 | 摘要要全面 |


## 什么是RAGAS 

RAGAS  全称为 RAG Assessment，RAG评估，是一个专门用于评估、监控和提升生产环境中大模型和检索生成应用的性能，包括用于生产质量监控的定制模型。

生成角度可以从忠实性（faithfulness）、回答相关性（Answer relevancy）评估。

检索角度可以从上下文精度（Context precision）、上下文召回率（Context recall）评估。

还有其他评估指标：
- 答案准确性（answer correctness）

- 上下文利用率（Context utilization）

- 上下文实体召回率（Context entity recall）

- 噪声敏感度（noise sensitivity）

评估流程：

1. 准备评估数据集，包含4字段：**question**、**answer**、**context**、**ground-truth**
注意：每一项都是一个数组列表，答案、上下文和基本事实和问题列表是一一对应的，即第一个问题的答案也必须是 answer 中的第一个元素，同时也必须是上下文和基本事实的第一个元素。其中上下文的每个元素都是一组字符串数组，这是因为每个问题都可以有多个上下文。

2. 开始评估
 - 加载评估数据集
 - 评估忠实性：需要用到 context、answer
    - Faithfulness 指标用于衡量答案的可信程度，即答案中的主张有多少是可以被 contexts 证实的。
- 回答相关性：需要用到 question、context、answer
    - Answer Relevance 指标用于衡量 RAG 生成的回答与问题之间的相关程度。此分数通过从答案中逆向推理变体问题，并计算与原始问题的余弦相似度，来衡量原始答案中的信息是否都与问题相关。

- 上下文精度：需要用到 question、ground——truth、contexts
    - Context Precision 用于衡量检索出的上下文与问题的相关程度，即同时衡量上下文与问题的相关性及排序情况。

- 上下文召回率：需要用到 ground_truth、contexts
    - 衡量检索到的上下文与作为基本事实的一致程度

## 两种主流方案评估

方案一：RAGAS（RAG Assessment）
- 开源库，专门评估RAG系统
- 输入：用户问题、检索到的文档、LL生成的答案
- 输出：Faithfulness、Answer Relevancy、Context Recall 等分数。
- 原理：内部使用另一个 LLM 作为评判器，自动打分
- 优点：标准化、可复现
- 缺点：依赖评判器质量，需要一定量级的数据才有统计意义。

方案二：LLM-as-Judge
- 思路：自己用LLM作为评判器
- 实现：写一个prompt，告诉LLM“请根据参考文档判断以下回答是否忠实，输出1-5分”
- 优点：灵活，可以自定义评估维度。
- 缺点：需要设计prompt，评分一致性可能不如 RAGAS。

两者关系：RAGAS 本质上也是 LLM-as-Judge。只是把prompt工程和统计工程封装好了。如果项目初期想要快速验证，可以使用RAGAS，如果想要深入了解或自定义指标，可以自己实现 LLM-as-Judge。


## 我的项目落地计划

### 1. 准备评估数据集
 - 从知识库（rag_samples + rag_medical）中抽取20-30个 有代表性的育儿问题。
 - 覆盖不同类别：常见病护理、喂养、睡眠、生长发育、疫苗与体检、日常护理。
 - 人工标注每条数据的期望答案要点，作为ground_truth

 ### 2. 构建评估脚本
  - 在 `script/` 下新建`eval_rag.py`
  - 流程：读取评估数据集 -》逐条调用 `/api/chat` 获取answer 和 references - 》组装评估数据 - 》调用 RAGAS 或者 LLM-asJudge 评分。

### 3. 评估指标选择
| 指标 | 衡量什么 | 为什么对我重要 |
|------|---------|-------------|
| Faithfulness（忠实度） | 回答是否基于检索文档，有无编造 | 育儿场景下错误信息后果严重 |
| Answer Relevancy（答案相关性） | 回答是否贴合用户问题 | 确保模型在“回答问题”而不是“泛泛而谈” |
| Context Recall（上下文召回率） | 检索是否找到了正确答案 | 检验混合检索相比纯 Dense 的提升 |

### 4. 预期产出
- 基线分数：当前 RAG 管道（混合检索 + DeepSeek）的三维评估分数
- 对比实验：纯 Dense vs 混合检索（线性加权 α=0.7）vs 混合检索（RRF）的评估分数对比
- 直接补上题库中 🟡 追问所需的定量数据（α 选值、融合策略对比、候选数量对比）


## 为什么优先选择三个核心维度

不是其他指标不重要，而是基于**实施成本和诊断效率**的权衡：

1. **低成本启动**：忠实度和答案相关性不需要 ground truth，仅上下文召回率需要人工标注。相比需要逐条标注标准答案的答案准确性，三维指标能在最短时间内跑通评估闭环。
2. **形成诊断链**：召回率 → 忠实度 → 答案相关性，三个指标形成完整的问题定位链，能快速判断问题出在检索还是生成环节。
3. **其他指标是进阶优化**：上下文精度、实体召回率、利用率等适合在基线达标后做精细调优时引入。

### 为什么核心三维是低成本启动的最佳选择

三个指标中：
- **Faithfulness（忠实度）**：完全不需要 ground truth，RAGAS 内部用 LLM 自动比对回答和检索文档，判断有无编造。
- **Answer Relevancy（答案相关性）**：同样不需要 ground truth，RAGAS 用 LLM 从回答中逆向生成变体问题，计算与原始问题的语义相似度。
- **Context Recall（上下文召回率）**：需要 ground truth，但只需要标注“回答这条问题应该包含哪些关键信息点”，不必写出完整标准答案，标注成本低。

三者结合，只需要对 20-30 条问题做轻量级标注（每条 30 秒），就能跑通完整的评估闭环。而 Answer Correctness 需要逐条标注完整标准答案，成本高出 5-10 倍，适合在基线达标后对高风险问题做精细评估时引入。


**我的落地策略**：第一轮评估用三维指标快速建立基线，后续根据瓶颈定位逐步引入其他指标做深度优化。
