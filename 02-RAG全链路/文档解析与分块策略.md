# 文档解析与分块策略

文档解析全流程：文档解析 -》数据提取-》数据清洗-》文档分块-》向量化嵌入

## 文档解析

**文档解析的核心是将格式各异、版式多样且包含多种元素的文档数据，如段落、表格、公式、标题、多列布局以及图片等文档区域，精准的转化为符合阅读顺序的字符串信息。**

### 为什么文档解析如此关键？

- 首先解析质量决定了向量检索的准确性。不准确或格式混乱的抽取会导致向量空间的语义偏差，从而削弱RAG的问答效果。
- 其次，许多传统工具只能处理“生成式”PDF文本，对于扫描件等非数字原生文档往往失效。


### 文档解析实现的方案

1. 基于规则方法
    - 依赖PDF的结构信息（页数、文本框、字体大小，位置等），通过手动设计规则去提取内容。（PyPDF2、Langchain Document Loaders 文档加载器）
2. 基于深度学习的方法
    - 结合图像分割、目标检测模型，对PDF进行版面分析，流程：将PDF渲染为图像 -》再用模型识别出页面中的“标题、段落、表格、图像”等区块 -》再使用OCR对每个区块进行文本识别 -》 结合位置、标签进行结构化重构
3. 基于多模态大模型的方法
    - 利用图文输入能力强大的模型（GPT-4V、Gemini、Claude、MiniGPT-4），直接输入页面截图或文档图片，由大模型进行内容提取和理解，流程：将PDF渲染成图片 -》提问如“请总结这页的主要内容”“这页的表格中第一列是什么含义” -》LMM 模型结合视觉+语言能力返回结构化或自然语言内容


| 选型建议维度 | 推荐工具 | 技术建议 |
|-------------|---------|---------|
| 文档类型 | Unstructured / Docling（数字化文档） | 优先用于 PDF、Word 等可复制文本；处理速度快，适配多格式，适合构建快速 pipeline |
| 文档类型 | DocXChain / DeepDoctection（扫描图片） | OCR + 布局解析效果佳；推荐用于合同扫描件、票据、照片类文档 |
| 多格式混合支持 | Docling / Unstructured | 适合文件格式复杂的项目，具备自动格式路由、统一输出接口等能力 |
| RAG 部署方式 | 离线：Docling / Unstructured<br>在线：DeepDoctection / DocXChain | 离线适合规模大、频次低数据；在线适合上传即问答，需优化容器性能和响应延迟 |
| 表格提取 | DocXChain / Docling | 推荐用于财报、报表等需保留结构的文档；支持输出 CSV / Markdown 等 |
| 性能优化 | 轻量框架 + 多进程 + 缓存 | 精简容器、禁用不必要模块、启用缓存或分布式处理提升效率 |
| 工具组合使用建议 | PyMuPDF + Docling<br>PaddleOCR + Unstructured | 多工具组合搭配能同时兼顾速度与解析准确度，推荐做前后级联处理流程 |


### 按格式选型

| 格式 | 生产级方案 | 说明 |
|------|-----------|------|
| PDF（文本型） | PyMuPDF (fitz) | 比 pdf.js 快 10x，保留段落/标题/字体信息，中文编码处理更好 |
| PDF（扫描型） | PyMuPDF 取图片 → OCR | 先判断是否有文本层，没有则每页渲染成图片丢给 OCR |
| PDF（混合型） | 文本层 + OCR 双路合并 | 有些 PDF 部分页是文本部分页是扫描 |
| 图片 | PaddleOCR / Tesseract | PaddleOCR 对中文识别率高，Tesseract 通用但中文弱 |
| Word (.docx) | python-docx / mammoth | docx 本质是 zip 包 XML，直接解析取段落+表格 |
| Excel (.xlsx) | openpyxl / pandas | 保留行列结构，转 Markdown 表格给 LLM 最友好 |
| PPT (.pptx) | python-pptx | 提取每页标题+正文 |
| HTML | BeautifulSoup + 去标签 | 保留 h1-h6 标题层级 |
| Markdown | 直接读取 | 本身就是结构化文本 |


### 最佳实践

**按照文档类型MIMI TYP，去做不同的文本解析，**

流程如下：按 MIME type 分发 → 抽取文本 + 保留结构 → 输出 Markdown 文本 

TXT    → 直接读    
PDF    → PyMuPDF(文本层) / OCR(扫描件)  
DOCX   → python-docx(标题+段落+表格) 
XLSX   → openpyxl → Markdown 表格        
HTML   → BeautifulSoup(保留h1-h6) 
IMG    → PaddleOCR  



## 数据清洗

1. 文本标准化

这一步的目的是为了降低文本的特征离散度。如果用户输入“RAG”，而文档里写的是“rag ”，不标准化就会导致向量相似度下降。

- 统一编码格式（UTF-8）
    - 落地实践：在读取windows系统的.txt或者老旧的.html时，极易出现 GBK 或 ISO-8859-1编码，工程上统一使用python的chardet库进行自动编码检测，然后强转为UTF-8，彻底杜绝后续链路出现 UnicodeDecodeError
- 转化为小写形式（可选）：
    - 落地实践： 使用 BeautifulSoup 剥离 HTML 标签，但要保留文本；使用正则表达式（如 [^\w\s\u4e00-\u9fa5]）批量剔除不可见的控制字符（如 \x00、\u200b）以及连串的特殊符号（如 +++++、======）。
- 规范数字表达式
    - 落地实践： 将大写中文数字（一、二、三）和繁体数字统一映射为阿拉伯数字（1、2、3），或将“2026年”这种时间词规范化。这能防止在涉及财务、年报的 RAG 检索时，因为数字表达不同而彻底漏掉核心数据。

2. 噪声过滤

这一步的目标是纯净化内容，如果把广告、版权信息也做进向量库中，严重影响向量空间，导致大模型在回答问题时莫名带上广告

- 移除无关符号
    - 落地实践： 针对页眉、页脚、版权声明（如 "Copyright © All Rights Reserved"）、以及社交媒体文末的“点赞、关注、转发”。在工程上，通常维护一个通用噪声黑名单（RegEx 库）进行正则匹配剔除。针对 PDF 页面顶部的页眉页脚，则通过坐标范围直接裁剪（Layout-based cropping）。

- 删除重复段落/句子
    - 落地实践： 文档解析时常常会因为分页产生高频重复的“章节名”或“公司LOGO标语”。通过计算相邻段落的 Jaccard 相似度 或建立 MD5 哈希表，将完全相同的重复文本块直接过滤，防止其无意义地占用检索窗口（Top-K）。

- 过滤过短或无意义的内容片段
    - 落地实践： 设定硬性阈值。例如：字数少于 5 个字、或者整句话全是符号（如 ...、---）的片段直接丢弃。诸如“用户评论：”、“未完待续”这类不带实际语义的信息，必须拦截在向量库之外。

- 处理乱码文本区域
    - 落地实践： 当 PDF 字体未嵌入或被加密时，解析出来全是 📂🔶🔲 或不连贯的无意义拼音。通过统计一个文本块中非中文字符/非正常英文字符的比例（超过 30% 视为乱码），直接触发预警或抛弃该片段，避免其干扰正常检索。

3. 结构化解析

这一步的目标是恢复语义脉络，RAG的核心是分块，而分块的依据完全取决于这一步理出来的“骨架”

- 提取标题层级关系
    - 落地实践： 不能把文档清洗成一长串没有排版的流水账。在清除 HTML/PDF 的原始样式时，必须将标题转化为 Markdown 格式（# 标题一、## 标题二）。这个层级关系不仅是后续“基于标题分块”的依据，更是“前置语义注入”（把上级标题拼接进每一个子分块中）的硬性前提。

- 分离正文与元数据字段
    - 落地实践： 在清洗的尾声，利用规则或大模型（如 LayoutLM）将文档的“元数据”（如：发布日期、作者、文档分类、密级）剥离出来，单独存储为一个 JSON 字典（Metadata），而不是混在正文里。这为后续向量数据库的 元数据过滤（Metadata Filtering） 打下完美基础。

- 解析表格、列表等复杂数据结构
    - 落地实践： 表格是 RAG 的头号杀手。绝对不能把表格当成纯文本直接读出来（会变成一堆错位的字）。目前工业界标准的清洗做法是：利用 Table Transformer 或 Marker 等布局识别工具，将所有表格统一转换为合法的 Markdown 表格代码（|---|---| 结构）或 HTML Table，这样大模型和 Embedding 才能看懂行列的对应关系。

- 保留原文档语义边界
    - 落地实践： 清洗时必须识别并保留自然的语义物理边界，如段落标识（\n\n）、句号（。）。确保后续进行“递归字符分块”时，算法能够准确找到最佳的切分点，而不会在清洗阶段就把标点符号错抹掉，导致长句粘连。



## 文档分块

### 什么是文档分块

文档数据经过解析后，通过分块技术将信息内容划分为适当大小的文档片段，从而使RAG系统能够高效处理和精准检索这些片段信息。分块的本质在于依据一定的逻辑或语义原则，将较长文本拆解为更小的单元。

### 分块的方式

1. 固定大小分块，按预设的字符数或者Tokens直接切分。
    - 优点：速度快，行为可预测，处理大规模、结构混乱的数据集时很实用
    - 缺点：语义被截断。
    - 实践中，一般通过在相邻块之间设置一定的重叠来缓解这个问题。（chunk_overlap、chunk_size），切分文本时，连续的分块之间通常会加入一小段重叠区域来维持上下文连贯。所谓的重叠就是前一个分块的尾部几句话，在下一个分块开头再出现一次。
    - 适用场景：日志文件、邮件、代码仓库，以及结构参差不齐的大型语料库。
    
2. 基于句子的分块

按照完整句子来划分文本，而不是任意长度一刀切，每个分块至少包含一句或者多句完整的句子，语法完整，语义连贯
- 优点：每个分块都是一个有意义的思想单元

3. 基于段落分块

以完整段落为单位切分，不再拘泥于单个句子或者Tokens数。

每个分块往往对应一个独立的主体或者子主题。

- 适用场景：长篇文档、研究论文、综述类文章

4. 语义分块

利用Embedding或相似度分数来识别文本中天然的断裂点 -- 主题切换、上下文转折、章节边界。产出的分块语义清晰度更高，边界和语义对齐，检索质量有明显提升。
- 适用场景：知识库、技术文档、结构化文章
- 缺点：计算开销较大、分块长度不一致后续处理需要额外考虑。

5. 递归分割

它是固定大小和语义分块之间的一种折中方案，核心思路：先尊重文档结构，只在必要时才进行进一步切分。

具体做法：先尝试按标题切分，如果某个章节还是太长，就按照段落切。句子仍然超限，最后才按字符兜底。这样得到的分块既保有语义完整性，尺寸也在可控范围内。

- 适用场景：开发者文档、技术手册、学术论文、研究报告

6. 滑动窗口分块
有些文本的语义天然是跨句分布的，法律合同、科学论文、长段论证，一个完整的意思可能需要跨好几个句子，滑动窗口就是为这个而设计的。它不生成彼此独立的分块，而是创建相互重叠的窗口。比如窗口大小400Token，每次滑动200Token，这样相邻的分块之间有一半是共享的，语义在边界处不会断裂。

**优点**上下文保持的好
**缺点：**，分块数量会膨胀，存储和检索成本都会上升。
**适用场景**，法律RAG、金融分析、医学文献检索、合规审查。

7. 层次化分块

它是一个多层级的架构：小分块负责细粒度精确检索，中等分块支撑平衡的原理，大分块维持全局上下文。

检索时，系统先用小分块锁定精确位置，再把关联的大分块拉进来补充完整上下文。


## 数据预处理策略

**数据预处理**：把数据切块、打包、贴标签并存入仓库。

1. 分块策略

参照上面的**文档分块** 模块

2. 向量化准备

这一步是 RAG 的核心技术纽带，负责将人类的自然语言转化为数学空间中的稠密向量。

- 使用Sentance-BERT / BGE 等模型生成嵌入向量

- 对查询语句同样执行 embedding 操作以便匹配

- 存储向量 + 对应原文构成知识库

   **落地实践**： 选用主流的向量数据库（如 Milvus、Pinecone 或 Qdrant）。数据库的每条记录（Entity）至少包含三个核心字段：id、vector（高维浮点数数组）和 payload/metadata（包含清洗后的原文、清洗阶段第3步抽出的标题层级等）。

3. 元信息增强

这一步是打通混合检索（Hybrid Search）和精确过滤的关键，是目前大厂 RAG 提升精度的标准配置。

- 提取关键词作为辅助检索信号
   **落地实践**： 在预处理时，利用 Jieba（基于 TF-IDF/TextRank）或调用轻量级大模型，为每个 Chunk 提取出 3-5 个核心关键词。这些关键词不仅可以作为标签存入向量库的 Metadata 中，还能直接参与底层传统搜索。
- 加入时间戳、来源分类信息
   **落地实践**： 结合清洗阶段分离出的元数据，为 Chunk 打上显式标签（如 {"year": 2026, "department": "finance", "source": "annual_report.pdf"}）。在用户提问时，如果用户说“帮我查一下2026年的财务数据”，系统可以先通过 元数据过滤（Metadata Filtering） 在数据库中强行秒级过滤掉非2026年的所有向量，再在剩下的安全范围内做向量相似度计算，检索准确率会发生质的飞跃。

- 构造倒排索引加快粗筛速度
    - 落地实践： 向量检索擅长处理语义泛化（如提问“开心”能检索到“快乐”），但极度不擅长处理人名、产品型号、专有名词（如“iPhone 17 Pro Max”）。因此，预处理时必须同步把清洗后的原文送入 Elasticsearch 或向量库自带的 BM25 引擎 建立倒排索引（Inverted Index）。
    - 检索时，BM25 关键-词检索（稀疏检索）与向量检索（稠密检索）多路并行，利用倒排索引完成超大吞吐量的秒级粗筛，最后用 Reranker（重排模型，如 BGE-Reranker） 将两路结果融合成最终的 Top-K 吐给大模型。
